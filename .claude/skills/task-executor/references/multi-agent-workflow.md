# 多 Agent 协作流程

## 概述

对于复杂任务（高难度、长时间、多文件），task-executor 采用多 Agent 协作模式，确保执行质量。

---

## 角色定义

### 1. Executor (task-executor)

**职责**：核心代码实现

**主要工作**：
- 深度解析 PROMPT.md 需求
- 分析项目上下文和约束
- 制定详细的实施计划
- 生成/修改代码实现
- 响应 Reviewer 的反馈
- 修复 Validator 发现的问题

**输出**：
- 代码实现
- 实施过程文档
- 变更说明

---

### 2. Reviewer (code-reviewer)

**职责**：代码审查、质量把关

**主要工作**：
- 审查代码实现是否符合 PROMPT 要求
- 检查是否遵循项目约束条件
- 识别潜在的 bug 和风险
- 评估代码质量和可维护性
- 提供改进建议

**审查清单**：
- [ ] 需求覆盖度：是否实现了所有要求
- [ ] 约束遵守：是否遵循了约束条件
- [ ] 代码质量：可读性、可维护性
- [ ] 类型安全：TypeScript 类型正确性
- [ ] 命名规范：符合项目约定
- [ ] 边界处理：异常和边界情况

**输出**：
- 审查报告
- 问题列表
- 改进建议

---

### 3. Validator (task-validator)

**职责**：运行测试、验证验收标准

**主要工作**：
- 运行 EVAL.ts 测试
- 对照 SCORING.md 逐项验证
- 检查验收标准达成情况
- 验证元数据更新

**验证清单**：
- [ ] EVAL.ts 全部通过
- [ ] 自评分数 >= 70 (通过线)
- [ ] 关键检查点无失败
- [ ] 无类型错误
- [ ] 无 lint 错误

**输出**：
- 测试报告
- 验证结果
- 评分详情

---

## 协作流程

### 标准流程

```
┌─────────────────────────────────────────────────────────────┐
│ Phase 0: 任务分发                                           │
│   Executor 接收任务，分析复杂度和 Agent 需求                │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Phase 1: Executor 实施                                      │
│   1. 分析 PROMPT.md 和项目上下文                            │
│   2. 制定实施计划                                           │
│   3. 生成/修改代码                                          │
│   4. 自测基本功能                                           │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Phase 2: Reviewer 审查                                      │
│   1. 阅读 PROMPT.md 理解需求                                │
│   2. 审查 Executor 的代码实现                               │
│   3. 填写审查清单                                           │
│   4. 输出审查报告                                           │
└─────────────────────────────────────────────────────────────┘
                              │
                    ┌─────────┴─────────┐
                    ↓                   ↓
              发现问题             审查通过
                    ↓                   ↓
              返回 Executor           进入 Phase 3
                    ↓
              修复问题 → 重新审查
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Phase 3: Validator 验证                                     │
│   1. 运行 EVAL.ts 测试                                      │
│   2. 对照 SCORING.md 评分                                   │
│   3. 验证验收标准                                           │
│   4. 输出验证报告                                           │
└─────────────────────────────────────────────────────────────┘
                              │
                    ┌─────────┴─────────┐
                    ↓                   ↓
              发现问题             验证通过
                    ↓                   ↓
              返回 Executor         进入 Phase 4
                    ↓
              修复问题 → 重新验证
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Phase 4: 结果汇总                                           │
│   1. 合并所有 Agent 的输出                                  │
│   2. 生成最终执行报告                                       │
│   3. 更新 task-metadata.yaml                                │
│   4. 标记任务完成                                           │
└─────────────────────────────────────────────────────────────┘
```

## 触发多 Agent 协作的条件

当任务满足以下任一条件时，自动启用多 Agent 模式：

| 条件 | 阈值 | 说明 |
|------|------|------|
| 预估时间 | > 60 分钟 | 复杂任务需要分工协作 |
| 难度等级 | hard | 高难度任务需要专家审查 |
| 文件修改数 | > 5 个 | 大范围修改需要多视角审查 |
| 跨模块依赖 | 有 | 影响面广，需要谨慎验证 |

## Agent 间通信协议

### 消息格式

```typescript
interface AgentMessage {
  from: 'executor' | 'reviewer' | 'validator';
  to: 'executor' | 'reviewer' | 'validator';
  type: 'review_request' | 'review_feedback' | 'validation_request' | 'validation_result' | 'fix_complete';
  payload: {
    taskName: string;
    data: any;
    timestamp: string;
  };
}
```

### 状态流转

```
Pending → ExecutorWorking → Reviewing → Validating → Completed
                              ↓            ↓
                          Rejected     Failed
                              ↓            ↓
                        ExecutorWorking (循环)
```

---

## 质量门禁

### Executor → Reviewer
- 代码必须能编译/解析通过
- 必须包含基本的自测
- 必须提供变更说明

### Reviewer → Validator
- 审查清单全部通过，或
- 遗留问题已记录且非阻塞

### Validator → Completed
- EVAL.ts 全部通过
- 自评分数 >= 70
- 关键检查点无失败

---

## 故障处理

### Reviewer 发现严重问题
1. 标记阻塞性问题
2. 返回 Executor 修复
3. 修复后重新走审查流程

### Validator 测试失败
1. 输出详细的失败信息
2. 返回 Executor 修复
3. 修复后重新走验证流程

### 循环修复超过 3 次
1. 触发人工介入
2. 重新评估任务分解
3. 必要时拆分任务

---

## 最佳实践

### Executor
- 实施前充分理解需求
- 保持代码提交粒度小
- 及时响应审查意见

### Reviewer
- 优先关注正确性而非风格
- 区分阻塞性和建议性问题
- 提供具体的改进建议

### Validator
- 确保测试环境干净
- 记录完整的测试输出
- 准确对照评分标准
