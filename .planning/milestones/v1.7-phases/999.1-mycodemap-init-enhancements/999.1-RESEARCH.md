# Phase 999.1 Research: mycodemap init reconciler

**Status:** Ready for planning

## Scope

Phase 999.1 upgrades `mycodemap init` from a root-config-file creator into a project-state reconciler for CodeMap-owned infrastructure.

Primary planning inputs:

- `.planning/phases/999.1-mycodemap-init-enhancements/999.1-CONTEXT.md`
- `docs/brainstorms/999.1-mycodemap-init-enhancements-requirements.md`
- `.planning/ROADMAP.md`
- `.planning/STATE.md`

The brainstorm requirements document is the latest product truth for this planning pass. Where it differs from the earlier context, prefer the brainstorm document because it resolves the "non-Git project" behavior and adds the lightweight managed-asset ledger requirements.

## Current Repo Facts

### Init is currently root-config centric

- `src/cli/commands/init.ts` only checks root `mycodemap.config.json` and legacy root `codemap.config.json`, then writes a default config.
- `src/cli/paths.ts` currently declares `CONFIG_FILE_NEW = 'mycodemap.config.json'` and returns root config paths from `resolveConfigPath()`.
- `src/cli/config-loader.ts` already centralizes config loading through `resolveConfigPath()`, so changing this seam covers most downstream commands without touching every command.
- `src/cli/first-run-guide.ts` already writes the marker under `.mycodemap/`, but first-run detection still looks for root config files.

### Workspace defaults are half migrated

- `createDefaultCodemapConfig()` defaults `output` to `.mycodemap`.
- `createDefaultStorageConfig()` still defaults `storage.outputPath` to `.codemap/storage`, so a new canonical workspace would still emit legacy storage paths unless this is updated deliberately.
- `resolveOutputDir()` reads configured output through `resolveConfigPath()`, so canonical config-path changes and output-dir changes must be planned together.

### Hook assets are not currently packaged as `.githooks/`

- `package.json` publishes `dist/`, `docs/`, `examples/`, `scripts/`, README, license, changelog, and schema files.
- `.githooks/` is not in the package `files` list, so a globally installed CLI cannot assume `.githooks/pre-commit` and `.githooks/commit-msg` exist at runtime.
- Existing `scripts/hooks/install-hooks.sh` uses repo-local `.githooks` plus `core.hookspath`; Phase 999.1 requires target-project `.git/hooks/` shim behavior instead.
- Best planning path: keep `.githooks/` as authoring truth, but add shipped hook templates under a packaged location such as `scripts/hooks/templates/` or explicitly add `.githooks/` to the package surface with pack validation.

### Test coverage needs new init-focused fixtures

- There are tests for `paths.ts`, `config-loader.ts`, docs guardrails, and many commands.
- No focused `initCommand` test currently covers migration, receipt writing, hook install choices, rules generation, or repeat-run idempotency.
- Existing Vitest infrastructure is sufficient; no new test framework is required.

### Docs guardrails will become part of the implementation blast radius

- README, setup docs, AI guide docs, and validation scripts still mention root `mycodemap.config.json`.
- `scripts/validate-docs.js` explicitly guards against old config-name drift today, but will need to learn the new canonical `.mycodemap/config.json` wording.
- Because AGENTS.md requires docs updates for config changes and CLI behavior changes, documentation synchronization cannot be deferred past the final plan.

## Planning Implications

1. **Change shared path seams first.** Update `resolveConfigPath()`, defaults, and tests before expanding `init`; otherwise `init` can create `.mycodemap/config.json` that downstream commands ignore.
2. **Do not touch every command manually.** Most commands should inherit the new config path through `loadCodemapConfig()` and `resolveOutputDir()`. Hardcoded docs/tests are the main blast radius.
3. **Make receipt semantics a first-class internal type.** Requirements R1-R6 and R40-R43 need stable asset statuses, manual actions, hashes/version hints, and rollback hints. This is too much to keep as ad-hoc console strings.
4. **Resolve the non-Git discrepancy in favor of the brainstorm doc.** Non-Git projects should still reconcile workspace and rules; hooks become `skipped` or `manual-action-needed`, not a total init failure.
5. **Ship hook templates intentionally.** Runtime copying from `.githooks/` is unsafe unless pack validation proves those files ship. Prefer packaged templates or package-file inclusion plus smoke tests.
6. **Keep AI context files team-owned.** The CLI may generate copy-paste snippets and mark manual actions, but it must not auto-edit `CLAUDE.md` or `AGENTS.md`.

## Recommended Plan Split

### Plan 999.1-01 — Canonical workspace and config seam

Make `.mycodemap/config.json` the canonical config path and align defaults/tests so downstream commands can read what `init` writes.

Recommended files:

- `src/cli/paths.ts`
- `src/cli/config-loader.ts`
- `src/cli/__tests__/paths.test.ts`
- `src/cli/__tests__/config-loader.test.ts`

Acceptance focus:

- Prefer `.mycodemap/config.json`.
- Migrate/read root config only through explicit init behavior, not silent runtime fallback.
- Default storage paths no longer point new users at `.codemap/storage`.

### Plan 999.1-02 — Init reconciler, receipt, and first-run UX

Refactor `init` into scan/preview/apply/receipt helpers so it can explain project state without exceeding function-size limits.

Recommended files:

- `src/cli/commands/init.ts`
- `src/cli/init/*.ts`
- `src/cli/first-run-guide.ts`
- `src/cli/commands/__tests__/init-command.test.ts`
- `src/cli/__tests__/first-run-guide.test.ts`

Acceptance focus:

- Asset statuses include `missing`, `already-synced`, `migrated`, `installed`, `conflict`, `manual-action-needed`, and `skipped`.
- `.mycodemap/status/init-last.json` records minimal asset-level facts and rollback hints.
- Init output no longer says only "初始化完成" when manual actions remain.

### Plan 999.1-03 — Hook install, shim merge, and gitignore reconciliation

Install or sync hook payloads without clobbering existing user hooks, and record the exact result in the receipt.

Recommended files:

- `src/cli/init/hooks.ts`
- `scripts/hooks/templates/pre-commit`
- `scripts/hooks/templates/commit-msg`
- `src/cli/commands/__tests__/init-hooks.test.ts`

Acceptance focus:

- `.mycodemap/hooks/` receives synced hook payloads.
- `.git/hooks/*` receives a shim that preserves user hook logic.
- Non-Git projects get hook status `skipped` or `manual-action-needed`.
- `.gitignore` gets `.mycodemap/logs/` exactly once.

### Plan 999.1-04 — Rules bundle and manual AI context snippets

Generate a small generic rules bundle under `.mycodemap/rules/` and provide explicit copy-paste snippets for `CLAUDE.md` / `AGENTS.md`.

Recommended files:

Acceptance focus:

- Drift detection can show an update preview.
- AI context files are never auto-edited.

### Plan 999.1-05 — Docs, guardrails, pack smoke, and end-to-end QA

Synchronize docs and validation scripts with the new init contract, then prove the packaged CLI can initialize temp projects.

## Validation Architecture

Phase 999.1 needs validation at five layers:

1. **Path/config unit tests:** `paths.ts` and `config-loader.ts` prove `.mycodemap/config.json` is canonical and legacy roots are not silently preferred.
2. **Init fixture tests:** Temporary projects exercise new project, legacy root config, repeated init, non-Git, existing hooks, and manual AI context paths.
3. **Hook/rule asset tests:** Verify generated hooks and rules are hash-compared, receipt statuses are correct, and user hooks are preserved by shim logic.
4. **Docs guardrail tests:** `scripts/validate-docs.js` and its tests enforce the new canonical wording and required AI-doc updates.
5. **Package smoke:** `npm pack` into a temp prefix, run `mycodemap init --yes` inside fixture repos, and inspect `.mycodemap/status/init-last.json`.

Failure rehearsals to require in PLAN.md:

- Existing root `mycodemap.config.json` migrates to `.mycodemap/config.json` but the old file is not deleted silently.
- Non-Git project reconciles workspace/rules and reports hooks as `skipped` or `manual-action-needed`.
- Existing user hook is preserved by shim behavior.
- Missing packaged hook template fails as `manual-action-needed` or a clear error, not a false `installed`.
- Repeated init returns `already-synced` rather than rewriting assets unnecessarily.
- Manual AI context action remains visible until user adds the snippet; no automatic context-file edit occurs.

## Risks for Planner

- Do not implement a generic sync subcommand; `init` rerun is the only synchronization surface in this phase.
- Do not introduce profiles or complex preset abstractions before asset-level reconciliation works.
- Do not weaken existing docs, hook, or validation guardrails just to make the new contract easier to land.
- Keep helper functions short enough for repo redlines; avoid a single giant `initCommand`.
- Be careful with published-package asset paths. A plan that works from the repo root but fails after `npm install -g` does not satisfy the phase.
