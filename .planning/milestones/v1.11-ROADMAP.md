# Roadmap: CodeMap

## Current Planning Surface

**Status:** Active milestone `v1.11 release-followup-hardening`; `Phase 38 codex-release-entry-surface` and `Phase 39 publish-polling-and-reporting` both completed on 2026-04-23, and the next active phase is `Phase 40 readiness gate evaluation`.

`v1.11` picks up the deferred `RELF-01~03` follow-ups after `v1.9` / `v1.10`: ship non-Claude runtime release entry surfaces, add optional publish-status polling, and evaluate whether release readiness should become a stronger gate. ## Phases

### Phase 38: Codex release entry surface

**Goal**: 为 Codex 收口首个 non-Claude release entry / wrapper，同时保持 `/release` 与 `docs/rules/release.md` 仍是唯一 authority chain。
**Depends on**: Nothing (first phase)
**Requirements**: `RELF-01`、`SAFE-01`、`SAFE-02`
**Status**: Complete on 2026-04-23
**Plans**: 1/1 plans complete

Plans:
- [x] **38-01** Define Codex-first release wrapper strategy

**Success Criteria** (what must be TRUE):
1. Codex release 入口被明确建模，但它只能 route / delegate 到现有 release authority，而不是形成 competing path
2. wrapper / entry surface 的 refusal、confirmation 或 routing 规则与现有 `/release` contract 不冲突
3. milestone 内不会触发真实 `npm publish`、tag、push 或 GitHub Release
4. concrete runtime target 已固定为 Codex，write scope 与 deferred runtime（如 Kimi）被记录清楚，便于后续 phase 实施

### Phase 39: Publish polling and reporting

**Goal**: 为现有 GitHub Actions 发布链补一个可选 snapshot-only polling / structured report 面，降低手动盯发布状态的成本。
**Depends on**: Phase 38
**Requirements**: `RELF-02`、`SAFE-01`、`SAFE-02`
**Status**: Complete on 2026-04-23
**Plans**: 1/1 plans complete

Plans:
- [x] **39-01** Add publish-status polling contract

**Success Criteria** (what must be TRUE):
1. 发布状态 polling 只做读取 / 汇报，不隐式触发 rerun、publish 或其他副作用
2. 输出结构化 report，能区分 pending / success / failure / unavailable / ambiguous 等状态
3. 人类与 AI docs 会把 polling 明确描述为 follow-up observability，而不是新的 release authority
4. 缺失 workflow run、权限不足或 API 不可达时，会给出显式 non-success truth

### Phase 40: Readiness gate evaluation

**Goal**: 评估 release readiness 是否适合接入 CI 或 pre-release gate，并把 hard gate / warn-only / fallback 边界写成可验证 contract。
**Depends on**: Phase 39
**Requirements**: `RELF-03`、`SAFE-02`、`SAFE-03`
**Status**: Complete on 2026-04-29
**Plans**: 1/1 plans complete

Plans:
- [x] **40-01** Evaluate readiness gate integration path

**Success Criteria** (what must be TRUE):
1. release readiness 是否进入 CI / pre-release gate 会被明确判定，而不是停留在模糊建议
2. 若建议接入 hard gate，必须同时说明 calibration、fallback 与 false-positive 处理边界
3. 若当前不适合 hard gate，需要明确保留为 warn-only / manual step 的原因
4. 真实发布动作仍保持在未来显式 `/release v1.9` 与双确认门之后

## Out of Scope at Milestone Start

- 在本 milestone 中执行真实 `/release v1.9`、`npm publish`、git tag、push 或 GitHub Release
- 重写 `scripts/release.sh`、GitHub Actions 发布逻辑，或让 wrapper 取代 `/release`
- 把 release readiness 直接升成 hard gate，而不先完成评估、校准与 fallback 说明
- 顺手回补与本 milestone 无关的 broader governance 扩围或 archive 重写
