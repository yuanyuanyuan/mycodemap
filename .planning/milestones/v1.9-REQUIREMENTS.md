## v1 Requirements

Requirements for `v1.9 release-governance-unification`.

### Release Contract

- [x] **REL-01**: 维护者可以通过一份权威 `/release` 流程理解 milestone closeout 与 npm release 的 1:1 绑定关系
- [x] **REL-02**: 维护者可以清楚看到 milestone 版本到 npm 版本的映射规则（例如 `v1.9` → `1.9.0`），并在 major 跳跃时收到明确警告
- [x] **REL-03**: `/release` 流程会在目标版本、工作区状态、分支状态和 milestone readiness 不满足时拒绝继续

### Safety Gates

- [x] **SAFE-01**: `/release` 在 milestone 归档摘要展示后，必须等待用户显式确认才继续进入发布准备
- [x] **SAFE-02**: `/release` 在执行 npm version、git tag 或远程 push 前，必须等待用户输入明确的 `y` / `Y`
- [x] **SAFE-03**: 发布文档必须说明 closeout、版本 bump、tag、push、版本冲突等失败场景的处理与回滚方式

### Documentation Integration

- [x] **DOC-01**: `docs/rules/release.md` 定义 `/release` 的触发条件、完整流程、确认门、错误处理和禁止事项
- [x] **DOC-02**: `docs/rules/deployment.md` 与 `docs/rules/pre-release-checklist.md` 同步 milestone / npm release 绑定、版本统一和 readiness 检查
- [x] **DOC-03**: `AGENTS.md` 与 `CLAUDE.md` 能把发布治理路由到权威 release 文档，同时不复制整份流程正文

### Orchestrator Skill

- [x] **SKILL-01**: `.claude/skills/release/SKILL.md` 提供 `/release v{X.Y}` 薄编排器，委托 GSD closeout、现有 release script 和 GitHub Actions
- [x] **SKILL-02**: release skill 明确标注 L3 边界、版本跳跃警告和双确认门，不允许 AI 自主发布

### Validation

- [x] **VAL-01**: release 文档、路由与 skill 更新后，docs guardrails 和 docs-sync 验证通过
- [x] **VAL-02**: milestone 交付物包含至少三个失败场景预演：无 active/completed milestone、工作区不干净、版本跳跃

## v2 Requirements

### Release Follow-ups

- **RELF-01**: 为非 Claude runtime 提供等价 release skill / command 包装
- **RELF-02**: 可选轮询 GitHub Actions 发布状态并输出结构化报告
- **RELF-03**: 在未来版本中评估是否把 release readiness 检查接入 CI 或 pre-release gate

## Out of Scope

| Feature | Reason |
|---------|--------|
| 本 milestone 内执行真实 npm publish / GitHub Release | 当前目标是定义和验证发布流程，不是发布当前工作树 |
| 重建 `scripts/release.sh` 或 GitHub Actions 发布逻辑 | `/release` 是薄编排器，必须复用现有工具链 |
| 跳过 GSD milestone closeout 直接 npm release | 用户决策要求 milestone 与 npm release 1:1 绑定 |
| 在仓库中写入 `NPM_TOKEN` 或其他发布密钥 | 发布密钥只能来自环境 / GitHub Secrets |
| 自动接受 major 版本跳跃 | `0.x → 1.x` 或类似跳跃必须显式提示并等待确认 |
