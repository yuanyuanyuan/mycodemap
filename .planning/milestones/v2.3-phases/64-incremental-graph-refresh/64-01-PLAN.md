<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| changed-input source → generate CLI | `git diff` 与显式 changed-files 都是外部输入，可能为空、越界、失真或不完整 |
| generate orchestration → repository/storage | CLI 不能直接信任 scope 结果，必须通过窄 repository/storage seam 落盘 |
| temp graph rebuild → persisted SQLite truth | 局部重算结果只有在 validation/parity 成功后才能替换当前 truth |
| built CLI subprocess → temp repo artifacts | e2e 通过真实文件系统和真实子进程触发 shipped path，避免 mock 掩盖失败模式 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-64-01 | T | changed-file sourcing | mitigate | 显式 changed-files 优先于 diff 推导，越界/不存在/空 scope 直接返回 `INCREMENTAL_SCOPE_UNRELIABLE` 或 `INCREMENTAL_SCOPE_EMPTY`，并要求 full generate |
| T-64-02 | I | invalidation boundary | mitigate | 在 `graph-helpers.ts` 强制 `2-hop`、双向、对 `INFERRED`/`AMBIGUOUS` 保守扩张，边界无法判定时返回 `INCREMENTAL_INVALIDATION_BOUNDARY_UNRESOLVED` |
| T-64-03 | D | SQLite writeback | mitigate | `SQLiteStorage.ts` 先保留单个 pre-refresh snapshot，再 transactionally replace slice；writeback 失败返回 `INCREMENTAL_WRITEBACK_FAILED`，不污染现有 truth |
| T-64-04 | R | refresh observability | mitigate | human/JSON 输出与持久化 metadata 都记录 `success|partial|failed`、slice counts、affected reasons 与稳定 diagnostics codes |
| T-64-05 | E | unsupported server surface | accept | `AnalysisHandler` 仍保持 `INCREMENTAL_UPDATE_NOT_SUPPORTED` / `REFRESH_NOT_SUPPORTED` 公共 gate；本 phase 不把未验证的 server incremental API 偷偷开放 |
</threat_model>
