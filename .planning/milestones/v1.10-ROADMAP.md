## Overview

这个 milestone 处理 `v1.8` 与 `v1.9` closeout 后仍留在 backlog 里的治理债：入口文档重复 drift 自动检测、ghost commands / validation trust 清理，以及 archive/live 身份边界收口。
目标不是新增治理系统，而是把这些问题继续收敛到现有 `docs:check`、`ci check-docs-sync` 与 planning surface，让维护者能稳定判断哪份文档是 live truth、哪些只是 historical archive。
本 milestone 保持 brownfield / governance-only 范围：不触发真实 `/release`、不实现非 Claude release wrapper，也不把 snippet-based guardrail 一次性升级成新的生成式治理平台。

## Milestone Summary

**Key Decisions:**

- 用户将下一轮 milestone 明确命名为“治理债清理”，并把焦点锁定在 governance debt 而不是 release follow-ups
- 本 milestone 跳过外部 research，直接以现有仓库 docs / guardrail / archive 事实定 scope
- phase 编号延续 `35-37`，不重置为 `1`
- 检测能力必须复用现有 docs guardrail / CI 入口，而不是新建治理中间层

**Issues To Resolve:**

- entry-doc duplicate drift 目前仍缺少自动检测 / 自审 backstop
- ghost commands、validation trust 与 authority routing 仍可能在 live docs 之间漂移
- archive/live 身份治理还不够强，历史工件仍可能被误判为 current truth

**Issues Deferred:**

- 真实 `/release v1.9` 执行、npm publish 与 GitHub Release
- 非 Claude runtime 的 release wrapper / command 包装
- GitHub Actions publish polling 与 release readiness 接入 CI
- 更大范围的结构化文档 inventory / generation 平台化尝试

**Technical Debt To Watch:**

- `scripts/validate-docs.js` 目前以 snippet guardrail 为主，扩展时要控制 false positives
- `gsd-sdk query init.new-milestone` 仍会返回 stale current milestone metadata；planning durable files 才是当前真相源
- 任何 archive identity 修复都不能重新把 archive 文档写成 second source of truth
