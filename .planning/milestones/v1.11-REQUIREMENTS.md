## v1 Requirements

### Release Follow-ups

Phase 38 concrete runtime choice: **Codex-first** via repo-local `.agents` skill path. Other non-Claude runtimes stay deferred unless the Codex pattern proves trivially reusable.

- [x] **RELF-01**: 维护者可以从至少一种 non-Claude runtime 进入 release flow，但该入口必须 route / delegate 到现有 `/release` authority chain，而不是形成 competing release path
- [x] **RELF-02**: 维护者可以获取 GitHub Actions 发布状态的 structured report / polling truth，而不是只能手动刷新网页或日志
- [x] **RELF-03**: 维护者可以明确判断 release readiness 是否适合进入 CI 或 pre-release gate，并看到 hard gate / warn-only / fallback 的边界说明

### Safety / Scope Guards

- [x] **SAFE-01**: 新增 wrapper、polling 或 gate docs 不会削弱 `docs/rules/release.md` 作为 authoritative release workflow 的地位
- [x] **SAFE-02**: 本 milestone 不执行真实 `npm publish`、git tag、push 或 GitHub Release；真实发布仍需未来显式 `/release v1.9` 与双确认门
- [x] **SAFE-03**: 若 readiness gate 仍存在 calibration、超窗或 false-positive 风险，文档与实现必须明确保留 `warn-only / fallback` truth，而不是伪装成 hard gate success

## Out of Scope

| Feature | Reason |
|---------|--------|
| 本 milestone 内执行真实 `/release v1.9` 或实际发布动作 | 发布仍是 L3 边界；当前目标是 release follow-up hardening，不是实际发版 |
| 重建 `scripts/release.sh` 或 GitHub Actions 发布逻辑 | follow-up 必须复用现有 toolchain，而不是另起一套发布系统 |
| 让 non-Claude wrapper 成为与 `/release` 并列的主入口 | 这会重新引入 authority drift，与 `v1.9` / `v1.10` 目标冲突 |
| 未经评估直接把 readiness 升成 hard gate | 当前 milestone 只负责评估与收口 truth，不直接跳过 calibration |
| 回头重做 broader governance cleanup 或 archive rewrite | 与当前 release follow-up milestone 无关 |
