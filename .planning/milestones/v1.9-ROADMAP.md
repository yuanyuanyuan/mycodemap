## Overview

这个 milestone 已将 milestone closeout 与 npm release 收敛成统一的 `/release` 发布治理流程。
交付目标不是立即执行真实发布，而是先把“规划归档 → 版本映射 → 双确认门 → 机械发布工具链”的权威路径写清楚，并提供一个薄编排器 skill，复用现有 GSD closeout、`scripts/release.sh` 与 GitHub Actions。
本 milestone 全程保持 L3 发布边界：AI 只准备、验证、展示与等待确认；不会在本 milestone 内自主执行版本号变更、tag 创建、远程推送或 npm 发布。
2026-04-22 的 milestone audit 曾发现 `docs/rules/pre-release-checklist.md` 保留 helper-first 竞争入口；`Phase 34` 已关闭该 authority drift，并刷新 closeout-ready 验证证据。

## Milestone Summary

**Key Decisions:**

- `v1.9` scope 直接来自 `/home/stark/.claude/plans/ticklish-sprouting-church.md`
- `/release` 是薄编排器，不重建 GSD closeout、release script 或 GitHub Actions
- 每个 milestone 与 npm release 采用 1:1 绑定，`vX.Y` 映射为 npm `X.Y.0`
- 发布路径必须包含两道确认门，并保持 L3 发布操作不可自主执行

**Issues Resolved:**

- milestone closeout 与 npm release 现在重新回到单一 authority chain：`docs/rules/release.md` 定义正文，`pre-release-checklist.md` 只保留 `/release` 主入口与受控手动例外
- `AGENTS.md` 的 L3 边界、`CLAUDE.md` 的 release 路由与 Claude `/release` skill 现在互相一致
- 维护者现在可以在 skill 和 rules 中看到 `0.5.2-beta.1 → 1.9.0` 的 major jump 警告
- `docs:check`、`docs:check:pre-release`、`check-docs-sync` 和 diff hygiene 共同证明 release surface 当前可验证
- `Phase 34` 将 `$gsd-complete-milestone` 命令示例与 checklist helper 表达统一回当前 GSD 约定

**Issues Deferred:**

- 实际发布 `v1.9.0` 到 npm / GitHub Releases
- 自动轮询 GitHub Actions 并作为阻断 gate
- 为非 Claude runtime 提供等价 `/release` skill 包装

**Technical Debt To Watch:**

- `gsd milestone.complete` / `audit-open` 的参数行为仍建议在未来真实 `/release` dogfood 时再次核对
- 当前 `package.json` 到 `1.9.0` 是 major 跳跃，未来真实发布仍必须在确认门中高亮
- release skill 不得把 `NPM_TOKEN` 或任何 secret 写入仓库
- `Phase 31` / `Phase 32` / `Phase 33` 仍缺 `*-VALIDATION.md`，这属于非阻断 Nyquist 历史债务
