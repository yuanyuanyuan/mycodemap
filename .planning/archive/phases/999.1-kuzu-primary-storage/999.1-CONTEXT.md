# Phase 999.1: KùzuDB 作为主要存储

## Phase Boundary

**Goal**: 完全使用图数据库 (KùzuDB) 作为主力存储，文件系统作为兜底/迁移选项

**From ROADMAP.md**:
> 当前 v1.2 已实现图数据库后端，但采用双存储并行模式。本项探讨将 KùzuDB 提升为唯一真实源，同时保留文件系统作为：
> 1. 降级兜底（KùzuDB 不可用时）
> 2. 导出/迁移格式（JSON 交换）
> 3. 调试/审计用途

**Success Criteria**:
- [ ] KùzuDB 成为 `auto` 和默认配置的首选存储
- [ ] 原生图模型实现（节点/边表 + Cypher 查询）
- [ ] KùzuDB 初始化失败时自动降级到文件系统
- [ ] 现有文件系统用户数据自动无缝迁移到 KùzuDB
- [ ] 文档和配置同步更新

---

## Implementation Decisions

### DEC-01: 主存储策略
**Decision**: KùzuDB 成为 `auto` 和默认配置的首选

**Rationale**:
- 原生图数据库更适合 CodeMap 的核心价值主张
- 支持复杂依赖分析的 Cypher 查询
- 更好的查询性能和可扩展性

**Implementation**:
- 修改 `StorageFactory.determineStorageType()` 方法
- 从 `return 'filesystem'` 改为优先尝试 KùzuDB
- 默认配置 `storage.type: 'auto'` 将选择 KùzuDB

---

### DEC-02: 存储模型演进
**Decision**: 迁移到原生图模型

**Rationale**:
- 当前 snapshot 模式将整个图序列化为 JSON，没有利用图数据库优势
- 原生模型支持更高效的增量更新和复杂查询
- 为未来功能（如跨版本依赖分析）奠定基础

**Implementation**:
- 重构 `KuzuDBStorage` 适配器
- 定义节点类型：`Module`, `Symbol`, `File`
- 定义关系类型：`IMPORTS`, `EXPORTS`, `DEPENDS_ON`, `CONTAINS`
- 使用 Cypher 查询替代内存中的图遍历

**Schema Design**:
```cypher
// 节点标签
(:Module {id, name, path, type})
(:Symbol {id, name, type, line, column})
(:File {id, path, lastModified})

// 关系类型
(:Module)-[:IMPORTS]->(:Module)
(:Module)-[:EXPORTS]->(:Symbol)
(:Symbol)-[:DEPENDS_ON]->(:Symbol)
(:File)-[:CONTAINS]->(:Module)
```

---

### DEC-03: 降级与容错机制
**Decision**: KùzuDB 初始化失败时 fallback 到文件系统

**Rationale**:
- 保持系统可用性，不因为 KùzuDB 问题导致完全失败
- 给用户提供平滑过渡路径
- 符合"文件系统作为保底"的设计理念

**Implementation**:
```typescript
// StorageFactory.ts
async determineStorageType(config: StorageConfig): Promise<StorageType> {
  // 优先尝试 KùzuDB
  const kuzuAvailable = await this.checkKuzuAvailability();
  if (kuzuAvailable) {
    return 'kuzudb';
  }

  // Fallback 到文件系统
  logger.warn('KùzuDB unavailable, falling back to filesystem');
  return 'filesystem';
}
```

**Error Handling**:
- 降级时记录 warning 日志
- 返回的存储实例标记为 `fallback: true`
- 用户可选择修复 KùzuDB 后重新初始化

---

### DEC-04: 数据迁移路径
**Decision**: 现有文件系统用户自动无缝迁移

**Rationale**:
- 避免用户手动迁移数据的负担
- 保持向后兼容性
- 鼓励用户无感知地切换到 KùzuDB

**Implementation**:
```typescript
// KuzuDBStorage.ts
async migrateFromFilesystem(fsStorage: FileSystemStorage): Promise<void> {
  const existingData = await fsStorage.loadCodeGraph();
  if (existingData) {
    await this.saveCodeGraph(existingData);
    // 可选：保留原始文件作为备份
    await fsStorage.backup();
  }
}
```

**Migration Flow**:
1. 检测到 `storage.type` 切换到 `kuzudb` 或 `auto` 选择 KùzuDB
2. 检查文件系统中是否存在现有数据
3. 如果存在且 KùzuDB 为空，执行迁移
4. 迁移成功后保留或备份原文件

---

### DEC-05: 性能验证策略
**Decision**: 不需要建立基准测试来验证切换收益

**Rationale**:
- 原生图模型 vs JSON snapshot 的性能优势是理论明确的
- 嵌入式 KùzuDB 的启动和查询性能已在小规模测试中验证
- 重点放在功能正确性和稳定性，而非基准数字

**Validation Approach**:
- 集成测试覆盖主要使用场景
- 手动验证大规模代码库（如本仓库）的性能
- 关注错误率和降级场景的正确性

## Existing Code Insights

### Current Architecture

**StorageFactory.ts** (`src/infrastructure/storage/StorageFactory.ts`):
- 当前 `auto` 策略硬编码返回 `filesystem`
- 支持 `filesystem` | `memory` | `kuzudb` | `auto` 四种类型
- 需要修改 `determineStorageType()` 方法

**KuzuDBStorage.ts** (`src/infrastructure/storage/adapters/KuzuDBStorage.ts`):
- 当前使用 snapshot 模式：整个图序列化为 JSON 存储
- 关键方法：`serializeCodeGraphSnapshot()` / `deserializeCodeGraphSnapshot()`
- 演进方向：原生节点/边表 + Cypher 查询

**FileSystemStorage.ts** (`src/infrastructure/storage/adapters/FileSystemStorage.ts`):
- 当前默认存储
- 需要提供迁移接口供 `KuzuDBStorage` 调用

### Migration Complexity

**Low Risk Areas**:
- `StorageFactory` 的策略修改
- Fallback 机制的错误处理

**Medium Risk Areas**:
- 数据迁移的并发安全性
- 降级时的状态一致性

**High Risk Areas**:
- 原生图模型的完整实现
- Cypher 查询与现有查询接口的兼容
