## 1. System Classification

**System Type:** Hybrid (Code Automation + Internal AI Evaluation Infrastructure)

**Description:**
This phase extends the existing `codemap agent-metrics` reporting path into a CI-consumable token-threshold gate for CodeMap maintainers and automation users. Good behavior means the system evaluates persisted or freshly generated report rows correctly, keeps no-threshold runs visible but non-blocking, blocks only when an explicit `--max-tokens-per-query` is provided and exceeded by any single row, and emits stable human/JSON verdicts that CI can act on directly.

**Critical Failure Modes:**
1. Uses grouped summaries or totals instead of single-row `estimatedTotalTokens`, causing false pass/fail decisions.
2. Blocks CI when no explicit threshold was provided, or hides the warn-only path so regressions are invisible.
3. Emits human and JSON verdicts that disagree with each other or with the final exit code.
4. Invents a fake default numeric threshold before baseline calibration exists.
5. Breaks the locked command-path split by leaking gate behavior into `token` or silently auto-running from explicit `report`.

**Industry Vertical:** developer tooling / DevEx / CI gate infrastructure

**User Population:** CodeMap maintainers, CI/release engineers, internal automation consumers, and advanced developers using CodeMap with AI agents in pre-merge or release workflows.

**Stakes Level:** High

**Output Consequence:** False positives block merges and waste engineering time; false negatives let token-cost regressions or misleading summaries pass through CI and erode trust in the tool.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|------|------|------|------|------|
| Single-row threshold correctness | Any explicit threshold is checked against each `rows[*].estimatedTotalTokens` entry; one offending row is enough to fail | Uses averages, totals, or grouped summaries that hide outliers | High | CI correctness |
| Warn-only vs blocking semantics | No-threshold runs remain visible, advisory, and exit `0`; explicit threshold flips to blocking mode | Warn path is silent, or no-threshold runs unexpectedly fail CI | High | Developer trust |
| Honest baseline posture | System states that there is no default calibrated threshold yet and highlights the worst row instead | Hidden default threshold or fake pass line implies certainty that does not exist | High | Policy integrity |
| CI-consumable output | Human and JSON outputs surface verdict, threshold context, worst row, and violations clearly | Output is verbose, ambiguous, or impossible for CI to interpret reliably | High | Automation safety |
| Reproducibility / low noise | Same persisted run produces the same verdict and summary shape | Flaky paths, unstable ordering, or mixed command semantics produce noisy gates | Medium | Operational stability |

### Known Failure Modes in This Domain

- Grouped averages look healthy while an individual query row quietly exceeds the declared threshold.
- Flaky measurement or command-path drift turns the gate into noise that teams stop trusting.
- Warn-only behavior becomes effectively invisible, so teams lose early regression signal.
- Required-check or exit-code misconfiguration blocks the wrong branch state or silently permits regressions.
- A fabricated default threshold creates false confidence and weakens future calibration work.

## 2. Framework Decision

**Selected Framework:** No new external AI framework; extend the existing TypeScript CLI + service + interface-contract architecture

**Version:** N/A for AI runtime framework; use the current repo stack (`commander`, `zod`, `vitest`, existing CLI/service modules)

**Rationale:**
Phase 77 is not a conversational agent, RAG flow, or multi-agent runtime. It is a deterministic CI gate layered on top of an already shipped `agent-metrics` measurement/report path. The existing codebase already has the exact seams needed: thin CLI command handlers, a canonical `AgentMetricsService`, an interface contract for JSON shape, and prior CI threshold patterns. Adding LangChain, LangGraph, OpenAI Agents SDK, or similar orchestration would increase complexity without solving a real Phase 77 problem.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph | Useful only if later phases require persistent cross-run orchestration; overkill for a deterministic threshold gate |
| LangChain | Adds abstraction for prompts/tools/chains that this phase does not use at runtime |
| OpenAI Agents SDK | Introduces model/runtime coupling where Phase 77 is intentionally model-agnostic |

**Vendor Lock-In Accepted:** No

### Entry Point Pattern
```ts
type GateVerdict = 'pass' | 'warn' | 'fail';

interface GateViolation {
  sampleId: string;
  queryType: string;
  commandSlug: string;
  estimatedTotalTokens: number;
}

interface GateResult {
  verdict: GateVerdict;
  warnOnly: boolean;
  threshold: number | null;
  maxRow: GateViolation | null;
  violations: GateViolation[];
}

function evaluateReportGate(
  report: AgentMetricsReportResult,
  maxTokensPerQuery?: number,
): GateResult {
  const rows = [...report.rows].sort((left, right) => right.estimatedTotalTokens - left.estimatedTotalTokens);
  const maxRow = rows[0] ?? null;

  if (typeof maxTokensPerQuery !== 'number') {
    return {
      verdict: 'warn',
      warnOnly: true,
      threshold: null,
      maxRow,
      violations: [],
    };
  }

  const violations = rows.filter((row) => row.estimatedTotalTokens > maxTokensPerQuery);
  return {
    verdict: violations.length > 0 ? 'fail' : 'pass',
    warnOnly: false,
    threshold: maxTokensPerQuery,
    maxRow,
    violations,
  };
}
```

### Key Abstractions
| Concept | What It Is | When You Use It |

2. Setting `process.exitCode` inside the service layer instead of at the CLI edge.
3. Silently auto-running from explicit `report`, which breaks the latest-run-only contract.
4. Hiding the no-threshold path behind a generic success summary instead of an explicit warn-only verdict.

## 4. Implementation Guidance

**Model Configuration:**
No runtime model call is required for Phase 77. The production path is deterministic TypeScript logic over persisted report data. Any LLM usage should be limited to optional offline eval harnesses such as Promptfoo review prompts, with temperature `0`, fixed schema assertions, and no effect on the CLI verdict path.

**Core Pattern:**
Keep `agent-metrics` as a thin command wrapper over `AgentMetricsService`. Build or load a report, evaluate gate state from sorted row-level `estimatedTotalTokens`, attach a concise verdict block to the report output, render the same verdict in human and JSON modes, and set the exit code only at the command boundary.

**Tool Use:**
Reuse existing CLI/output helpers, the `AgentMetricsService`, the interface-contract module, and CI gate precedents from `src/cli/commands/ci.ts`. Use `Vitest` for deterministic logic tests, interface-contract asserts for schema stability, Promptfoo for human/JSON regression coverage, and Phoenix spans for observability if tracing is enabled.

**State Management:**
Persisted `agent-metrics` runs remain the source of truth. Explicit `report` must require an existing persisted run. Bare-root may reuse the latest run or auto-run measurement first, then evaluate the gate immediately. Gate metadata should be derived from the report object rather than stored as a separate durable entity in Phase 77.

**Context Window Strategy:**
The runtime path has no LLM context-window concern because it is deterministic. For optional eval harness prompts, pass only the fixture input, expected verdict context, and compact output snapshots; do not stuff full raw payloads or unrelated repository context into a judge prompt.

## 4b. AI Systems Best Practices

> Cross-cutting patterns every developer building AI systems needs, adapted to this phase.

### Structured Outputs with Pydantic

Use structured output contracts even when the production implementation is in TypeScript. For Phase 77, the runtime source of truth should stay in TypeScript types and `zod`, but this Pydantic model is a useful reference for eval harnesses, fixture validation, or any future judge-generated summaries.

```python
from pydantic import BaseModel, Field, ValidationError
from typing import Literal


class GateViolation(BaseModel):
    sample_id: str
    query_type: str
    command_slug: str
    estimated_total_tokens: int = Field(ge=0)


class GateVerdict(BaseModel):
    verdict: Literal["pass", "warn", "fail"]
    warn_only: bool
    threshold: int | None = Field(default=None, ge=0)
    max_row_tokens: int | None = Field(default=None, ge=0)
    violation_count: int = Field(ge=0)
    violations: list[GateViolation] = Field(default_factory=list)


def parse_gate_verdict(payload: dict) -> GateVerdict:
    try:
        return GateVerdict.model_validate(payload)
    except ValidationError as error:
        # Retry pattern for judge/eval flows:
        # 1. Surface the schema error
        # 2. Re-ask with the exact schema and failing fields
        # 3. Hard-fail if the second attempt is still invalid
        raise error
```

Current implementation guidance: mirror this structure with TypeScript interfaces plus `zod` schema checks before emitting JSON.

### Async-First Design

Keep the CLI and service path fully async, but keep business logic pure where possible. The common mistake here is mixing async I/O, rendering, and exit-code mutation in one function, which makes failures hard to test. Await storage/report operations in the service layer, keep gate evaluation synchronous and side-effect free, and reserve streaming for future long-running measurement UX rather than the threshold evaluation itself.

### Prompt Engineering Discipline

Phase 77 does not depend on prompt quality in production, but any eval-judge prompts should still separate system instructions from test-case payloads, pin expected schema, and keep examples minimal. Do not let a judge invent policy: the prompt should restate the locked rules such as "no threshold means warn-only" and "any row over threshold fails."

### Context Window Management

For eval prompts, include only the row subset needed to explain the verdict, plus the threshold and expected command path. Avoid sending the full repository, full raw measurement payload, or irrelevant grouped summaries. If a human-readable summary is being judged, prefer worst-row plus violating rows rather than entire result dumps.

### Cost and Latency Budget

Runtime budget: zero model calls, so latency should stay dominated by report loading or optional bare-root auto-run measurement. Eval budget: keep Promptfoo cases focused on a small fixture set in PRs and a larger reference set before release. Use temperature `0`, cache reusable fixtures, and avoid expensive judge passes on unchanged snapshot cases.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| CI decision closure | Pass: human and JSON outputs expose a clear gate verdict and CI can decide continue/warn/fail directly. Fail: command finishes without a usable verdict or exit semantics. | Code + Human | Critical |
| Threshold correctness | Pass: explicit threshold compares each `rows[*].estimatedTotalTokens`; `== threshold` passes and any `>` fails. Fail: averages, totals, or wrong fields drive the gate. | Code | Critical |
| Gate policy safety | Pass: no-threshold stays visible warn-only with exit `0`; explicit threshold alone can block. Fail: hidden warn path, default blocking, or fake default threshold. | Code + Human | Critical |
| Command-path and exit-code correctness | Pass: `token` stays pure measurement, bare-root may auto-run, explicit `report` missing run returns remediation error, and exit code matches verdict. Fail: path semantics drift or verdict/exit mismatch. | Code | Critical |
| JSON schema stability | Pass: report JSON keeps stable `schemaVersion` and adds a machine-readable gate block. Fail: consumers cannot reliably distinguish `pass` / `warn` / `fail`. | Code | High |
| Regression signal usefulness | Pass: output names the worst row, violations, threshold context, and query counts concisely. Fail: output is noisy or cannot answer which query exceeded the line. | Code + Human | High |

### Reference Dataset

**Size:** 12-example dev fixture set, expanded to 20 examples for release/pre-merge reference coverage

**Composition:**
- Threshold edges: below / equal / above / multiple violations
- Command-path cases: bare-root latest-run, bare-root auto-run, explicit report missing run
- Output cases: JSON pass / warn / fail parity, human summary visibility, verdict/exit-code parity
- Error cases: empty run, sample failure, schema drift, grouped-average false comfort, large-number formatting

**Labeling:**
Automatic truth labeling from fixture rows and expected command semantics. Human review is limited to summary clarity and remediation usefulness, with CI/release and maintainer spot checks on a small sample whenever schema or wording changes.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Force warn-only when no explicit threshold exists | `--max-tokens-per-query` absent on report/root path | Flag with visible warn verdict, highlight worst row, keep exit `0` |
| Fail on any row above explicit threshold | `rows[*].estimatedTotalTokens > threshold` | Block with `fail` verdict, include violations, set non-zero exit code |
| Keep `token` free of gate semantics | Threshold-like behavior attempted on `token` path | Reject unsupported usage or keep token path measurement-only |
| Preserve latest-run-only explicit report contract | `codemap agent-metrics report` called without persisted run | Block with remediation error; do not auto-run silently |
| Assert JSON gate block before emission | `--json` path just before output | Block on schema failure and exit non-zero rather than emitting partial JSON |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| 20 reference fixtures replay | Every PR | Block merge on threshold, path, or verdict regression |
| Human/JSON golden snapshot parity | Every PR | Inspect diff and update only with explicit policy approval |
| Max-row token distribution and fail-rate trend | Weekly | Investigate real regressions; do not paper over by inventing defaults |
| Warn/fail sample manual review | 5 samples weekly | Tighten summaries or fixture coverage if outputs stop being actionable |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix

**Key Metrics to Track:**
- `agent_metrics_gate_runs_total{verdict=error}`
- `agent_metrics_gate_json_schema_failures_total`
- `agent_metrics_gate_missing_run_total{command=report}`
- `agent_metrics_gate_auto_run_fallback_ratio{command=root}`
- `agent_metrics_gate_fail_rate{hasExplicitThreshold=true}`
- `agent_metrics_gate_max_row_tokens`
- `agent_metrics_gate_duration_ms`
- `agent_metrics_gate_verdict_exit_mismatch_total`

**Alert Thresholds:**
- Any JSON schema failure or verdict/exit mismatch should alert immediately.
- Any `verdict=error` on main/default branch within 24 hours should page the maintainer owning the gate.
- A sudden fail-rate jump above 20 percentage points for explicit-threshold runs should trigger regression investigation.
- Root auto-run fallback ratio above 30% over 7 days should trigger pipeline-health review.
- Duration p95 above 90 seconds should trigger measurement-path performance investigation.

**Smart Sampling Strategy:**
- Sample 100% of PR CI and main-branch gate runs.
- Sample 20% of local manual runs, or 100% when an explicit tracing env var is enabled.
- Retain 100% of `warn` / `fail` / `error` summaries plus violation metadata.
- Retain compact summary metrics for `pass` runs instead of full raw payloads.
