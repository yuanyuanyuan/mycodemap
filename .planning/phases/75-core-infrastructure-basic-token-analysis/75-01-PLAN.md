<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| fixed sample definitions -> token metrics truth | If the sample set is unstable or silently sourced from git history, repeated runs stop being comparable |
| serialized response truth -> estimated token counts | Users may mistake estimated values for exact tokenizer truth unless the contract is explicit |
| service persistence -> report rendering | Missing or malformed run/detail rows can produce misleading “successful” empty reports |
| new command family -> repo CLI surface | A custom output path or missing contract entry creates immediate drift from shared CLI conventions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-75-01 | Integrity | sample selection | mitigate | Keep a fixed built-in sample set in code and test its identities explicitly; defer git-history extraction |
| T-75-02 | Integrity | token estimation contract | mitigate | Use explicit `estimated*` field names, always pair them with `rawCharCount` / `responseSizeBytes`, and prove this in tests |
| T-75-03 | Reliability | persistence/report flow | mitigate | Fail loudly when no run or failed sample data exists; do not emit empty-success reports |
| T-75-04 | Scope creep | command/report implementation | mitigate | Exclude threshold gates, percentiles, trends, telemetry, and custom samples from the plan and tests |
</threat_model>
