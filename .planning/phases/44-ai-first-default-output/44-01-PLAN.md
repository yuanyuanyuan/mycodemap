    2. Create `src/cli/output/mode.ts`:
       - `resolveOutputMode(options: OutputModeOptions, ttyOverride?: boolean): OutputMode`
       - Logic: `--json` flag -> 'json'; `--human` flag -> 'human'; neither -> TTY auto-detect (`ttyOverride ?? process.stdout.isTTY` ? 'human' : 'json')

       - Export the function and the OutputMode type

    3. Create `src/cli/output/progress.ts`:
       - `createProgressEmitter(mode: OutputMode, prefix?: string): ProgressEmitter`
       - In 'json' mode: write `{"type":"progress","percent":50,"message":"halfway"}` lines to `process.stderr` using `process.stderr.write(JSON.stringify(...) + '\n')`
       - In 'human' mode: use `ora({ spinner: 'dots', prefixText: prefix })` for ora spinner. Call `spinner.text = message` on update, `spinner.succeed()` on complete, `spinner.fail()` on fail
       - In 'json' mode, `complete()` writes `{type:"progress", percent:100, message:"complete"}`
       - In 'json' mode, `fail()` writes `{type:"progress", percent:-1, message:msg}` or `{type:"error",...}`

    4. Create `src/cli/output/errors.ts`:
       - `formatError(error: unknown, mode: OutputMode): string | StructuredError`
       - For 'json' mode: return `JSON.stringify({type:'error', code: err.code ?? 'UNKNOWN', message: err.message, remediation: err.remediation})`
       - For 'human' mode: return `chalk.red('Error: ') + err.message + (err.remediation ? '\n  ' + chalk.yellow('Suggestion: ') + err.remediation : '')`
       - Handle non-Error thrown values gracefully (wrap in generic error)

    5. Create `src/cli/output/render.ts`:
       - `renderOutput<T>(data: T, humanRenderer: (data: T) => string, mode: OutputMode): void`
       - In 'json' mode: `process.stdout.write(JSON.stringify(data) + '\n')`
       - In 'human' mode: `process.stdout.write(humanRenderer(data) + '\n')`
       - Write directly to `process.stdout.write`, NOT `console.log` (avoids extra formatting; console.log adds newline but is fine too)

    6. Create `src/cli/output/index.ts` barrel: export everything from mode, progress, errors, render, types

    7. Create test files under `src/cli/output/__tests__/`:
       - `mode.test.ts`: Test resolveOutputMode with all flag combinations + TTY states
       - `progress.test.ts`: Test createProgressEmitter in both modes; spy on process.stderr.write for JSON mode; verify ora is not invoked in JSON mode
       - `errors.test.ts`: Test formatError with Error objects, non-Error values, and both modes
       - `render.test.ts`: Test renderOutput writes to process.stdout in both modes; spy on process.stdout.write

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI stdin/stdout pipe | Untrusted consumer reads stdout; output format determines parseability |
| CLI stderr pipe | Progress/events channel; must not contaminate stdout data stream |
| process.stdout.isTTY | Environment signal that can be spoofed; not a security boundary but a UX heuristic |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-01 | Tampering | resolveOutputMode | accept | isTTY spoofing only affects output format, not data integrity — low risk |
| T-44-02 | Information Disclosure | stderr progress events | accept | Progress messages are non-sensitive operational info; no PII |
| T-44-03 | Denial of Service | createProgressEmitter JSON mode | mitigate | Rate-limit progress writes: only emit when percent changes by >= 5% or message changes |
| T-44-04 | Tampering | formatError remediation field | accept | Remediation strings are static suggestions, not executable — no injection risk |
</threat_model>
