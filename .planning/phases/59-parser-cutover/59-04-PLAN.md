<objective>
Add DEPRECATED_PARSER_MODE rejection for deprecated parser modes (fast/smart/hybrid) on the server side, aligning with the CLI-side error handling from 59-01.
</objective>

<gap_source>
UAT Test 10: Server request types accept deprecated mode values for compatibility but reject them at runtime with DEPRECATED_PARSER_MODE.

Current behavior: AnalyzeRequest.options.mode type accepts 'fast'|'smart'|'hybrid', but analyze() throws 501 UnsupportedAnalysisOperationError for ALL requests — no specific DEPRECATED_PARSER_MODE rejection path exists for server-side deprecated modes.
</gap_source>

<must_haves>
- AnalyzeRequest.options.mode type still accepts 'tree-sitter' | 'fast' | 'smart' | 'hybrid' for backward compatibility
- When a deprecated mode (fast/smart/hybrid) is passed via the analyze endpoint, it is rejected with a DEPRECATED_PARSER_MODE error (400 status) containing remediation guidance, NOT a generic 501
- When tree-sitter mode (or no mode) is passed, the existing 501 UnsupportedAnalysisOperationError still applies (since analyze is not yet a supported operation)
- API route /analysis validates mode and returns 400 with DEPRECATED_PARSER_MODE code and nextCommand guidance
- Tests cover: deprecated mode → 400 + DEPRECATED_PARSER_MODE, tree-sitter/no mode → 501 + ANALYSIS_NOT_SUPPORTED
</must_haves>
