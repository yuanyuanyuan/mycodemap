# Phase 55: Agent Bootstrap Assets - Context

**Gathered:** 2026-05-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 55 delivers assistant bootstrap assets generated by `mycodemap init`.
It makes a newly initialized project easier for Claude, Codex, and generic AI
assistants to connect to CodeMap's project-local context without requiring the
tool to rewrite team-owned entry files.

In scope:
- Generate `.mycodemap/env-contract.json` as the initial Project Environment
  Contract seed, sourced from the selected bootstrap profile and obvious
  project manifests.
- Generate `.mycodemap/assistants/` with per-runtime bootstrap assets:
  - `claude-context.md` for project-root `CLAUDE.md` copy-paste guidance.
  - `agents-context.md` for project-root `AGENTS.md` copy-paste guidance.
  - `claude-hook-example.json` as an inactive Claude Code `SubagentStart`
    hook example.
  - `codex-agent-example.toml` as an inactive Codex agent
    `developer_instructions` example.
- Add `--assistant-profile claude|codex|generic` to select assistant asset
  generation independently from the existing project `--profile`.
- Make `mycodemap init --json` return the real machine-readable
  `InitReceipt` JSON.

Out of scope:
- Automatically rewriting project-root `CLAUDE.md`, `AGENTS.md`,
  `.claude/settings.json`, or Codex agent config files.
- Implementing Phase 58's full env-contract discovery engine, role filtering,
  conflict detection, doctor integration, or `mycodemap env-contract` command.
- Runtime/session CLI-first enforcement or subagent compliance auditing.
- Deep command inference from README, CI workflows, Makefile, or arbitrary docs.

</domain>

<decisions>
## Implementation Decisions

### Assistant Profile Flag Semantics

- **D-01:** Add a new `--assistant-profile claude|codex|generic` flag for
  assistant runtime asset selection. Do not reuse existing `--profile` for this
  purpose.
- **D-02:** Keep `--profile nodejs|python|go|rust|generic` as the project
  bootstrap profile selector from Phase 53.
- **D-03:** Allow `--profile` and `--assistant-profile` to be combined. Example:
  `mycodemap init --profile nodejs --assistant-profile codex -y` means "use the
  Node.js project profile and generate Codex assistant assets."
- **D-04:** If `--assistant-profile` is omitted, generate all assistant assets
  for `claude`, `codex`, and `generic`. Do not infer the target runtime from
  `.claude/`, `.codex/`, or other environment markers in this phase.

### env-contract.json v1 Content Boundary

- **D-05:** `.mycodemap/env-contract.json` uses a typed seed contract, not a
  flat JSON object and not the full Phase 58 contract. It should include
  `schemaVersion`, `generatedAt`, `projectProfile`, and `items[]`.
- **D-06:** Each contract item should carry at least `category`, `key`, `value`
  or `status`, `source`, and confidence/status metadata. This keeps the file
  future-compatible with Phase 58 without implementing Phase 58 now.
- **D-07:** Phase 55 contract sources are limited to the selected bootstrap
  profile plus obvious project manifests. Examples: `package.json` scripts,
  `pyproject.toml`, `go.mod`, and `Cargo.toml` basic facts. Do not scan README,
  CI, Makefile, docs, or hooks for deep discovery in this phase.
- **D-08:** Missing items must be represented explicitly as
  `status: "unknown"` with a source such as `not-detected`. Do not omit checked
  items silently and do not invent profile defaults for unknown commands.

### Assistant Asset Depth

- **D-09:** `claude-context.md` and `agents-context.md` are short entry snippets
  plus routing maps. They are suitable for copy-paste into `CLAUDE.md` and
  `AGENTS.md`, but they are not full replacement templates.
- **D-10:** Main-agent context files should point to `.mycodemap/env-contract.json`,
  `.mycodemap/rules/`, `.mycodemap/status/init-last.json`, and relevant CodeMap
  commands such as `mycodemap doctor` / `mycodemap generate`. They should not
  duplicate full rule text.
- **D-11:** `claude-hook-example.json` and `codex-agent-example.toml` are
  copyable examples that are inactive by default. They should show how a user
  can make subagents retrieve the env contract, but init must not write them
  into platform-owned configuration paths.
- **D-12:** Assistant context files include a short project-specific summary:
  detected project type/profile, env-contract path, rules path, init receipt
  path, and any unknown contract item warnings. Do not inline the full
  env-contract JSON.

### init --json Output Contract

- **D-13:** `mycodemap init --json` writes the complete `InitReceipt` structure
  to stdout as the single machine-readable JSON document.
- **D-14:** In `--json` mode, stdout must contain JSON only. Do not print human
  preview/receipt text to stdout. Warnings and errors may go to stderr.
- **D-15:** Preview mode remains side-effect-free. `mycodemap init --json`
  without `-y/--yes` returns a preview receipt on stdout but does not write
  `.mycodemap/status/init-last.json` or other workspace files.
- **D-16:** Apply mode (`-y/--yes`) continues to write the workspace and
  `.mycodemap/status/init-last.json`; `--json` only changes the stdout renderer.

### Agent's Discretion

- Exact TypeScript module layout for assistant asset generation is a planner
  choice. A dedicated `src/cli/init/assistants.ts` module is likely cleaner than
  expanding `rules.ts`, but this is not locked.
- Exact env-contract schema field names are planner discretion as long as the
  D-05 through D-08 constraints are preserved.
- Exact human wording in assistant context files is planner discretion as long
  as files stay short, copyable, and route to canonical local assets.
- Exact list of obvious manifest extractors is planner discretion, bounded by
  D-07.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `InitAsset` / `InitReceipt` already represent install state, ownership,
  details, hashes, rollback hints, manual actions, summary counts, and next
  steps. Assistant assets should reuse this model instead of introducing a
  parallel receipt shape.
- `createInitPlan()` already aggregates workspace, config, hook, rules, profile,
  and first-run assets. Assistant and env-contract plans can be added as another
  asset family.
- `applyInitPlan()` already centralizes apply-mode writes. New writes should
  stay behind apply mode unless they are preview-only in memory.
- `createRulesPlan()` and `aiContextAsset()` already demonstrate the correct
  pattern for team-owned AI context files: detect existing references, otherwise
  emit `manual-action-needed` copy-paste guidance without editing files.
- `createProfilePlan()` already models profile selection as an init asset and
  can provide project profile data to the env-contract seed generator.
- `loadProfile()` / `detectProjectType()` provide the project profile facts
  Phase 55 needs.

### Established Patterns

- Preview is default; apply requires `-y/--yes`.
- Tool-owned files under `.mycodemap/` may be written by init in apply mode.
- Team-owned files such as `CLAUDE.md` and `AGENTS.md` are never silently
  rewritten.
- Receipt output is the truth source for installed, skipped, already-synced,
  conflict, and manual-action-needed states.
- Public CLI changes require commander registration, interface-contract sync,
  docs sync, and focused tests.

### Integration Points

- Add assistant option parsing in `src/cli/index.ts` and `src/cli/commands/init.ts`.
- Add assistant/env-contract asset planning into `src/cli/init/reconciler.ts` or
  a new helper module consumed from there.
- Generate files under `.mycodemap/assistants/` and `.mycodemap/env-contract.json`
  only in apply mode.
- Update `src/cli/interface-contract/commands/init.ts` so `--json`,
  `--assistant-profile`, and the `InitReceipt` output shape match runtime.
- Add tests that cover:
  - default all-assistant generation;
  - `--assistant-profile codex` subset generation;
  - project `--profile` + assistant `--assistant-profile` combination;
  - JSON stdout parseability with no human stdout;
  - preview mode no filesystem writes;
  - apply mode receipt includes assistant and env-contract assets.

### Current Gaps

- `init` commander registration currently lacks `--json` despite the interface
  contract advertising it.
- `InitCommandOptions` currently has no `json` or `assistantProfile` field.
- `executeInitCommand()` currently always prints human text before returning the
  receipt.
- Existing AI context snippet is rules-only and receipt-embedded; Phase 55 needs
  actual `.mycodemap/assistants/` files.
- No `.mycodemap/env-contract.json` writer exists yet.

</code_context>

<specifics>
## Specific Ideas

Env-contract seed shape sketch:

```json
{
  "schemaVersion": "env-contract.seed.v1",
  "generatedAt": "2026-05-02T14:15:47+08:00",
  "projectProfile": {
    "name": "nodejs",
    "source": "package.json",
    "confidence": "high"
  },
  "items": [
    {
      "category": "execution",
      "key": "testCommand",
      "value": "npm test",
      "source": "package.json:scripts.test",
      "confidence": "high"
    },
    {
      "category": "execution",
      "key": "buildOutput",
      "status": "unknown",
      "source": "not-detected",
      "confidence": "none"
    }
  ]
}
```

Assistant asset behavior:
- `--assistant-profile codex` generates only Codex-specific assets plus any
  shared generic files required for the selected output set.
- No flag generates all four roadmap-listed files.
- `claude-context.md` and `agents-context.md` should read like copy-paste
  snippets, not canonical policy documents.
- Adapter examples should say "copy this into your platform config" rather than
  pretending they are already active.

</specifics>
