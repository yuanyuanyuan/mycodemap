# Phase 52: CodeMap CLI Priority Harness Guard - Context

**Gathered:** 2026-05-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 52 owns runtime/session-level compliance for the CodeMap CLI-first retrieval rule. It is separate from Phase 51: Phase 51 improves `mycodemap init` post-install bootstrap assets, while Phase 52 makes actual Claude/Codex sessions observable when they bypass `mycodemap query|analyze|deps|impact` for code search, project analysis, or impact analysis.

In scope:
- Clarify the machine-checkable CLI-first gate in repository governance docs.
- Add report-only guard or audit tooling for non-compliant first-use `rg` / `grep` / `find` in code paths.
- Preserve documented fallback cases for docs/config/log searches, CLI failure, unsupported paths, and insufficient CLI results.
- Verify failure, compliant, and allowed-fallback scenarios using real filesystem + subprocess evidence.

Out of scope:
- New-user post-install bootstrap assets generated by `mycodemap init`; this belongs to Phase 51.
- Silent hard-blocking before report-only evidence exists.
- Global enforcement across every external AI runtime.

</domain>

<decisions>
## Implementation Decisions

### Phase Split
- **D-01:** Keep this phase focused on session/runtime compliance. Do not fold post-install initialization work into this phase.
- **D-02:** Treat Phase 51 as an upstream setup improvement, not as a substitute for runtime observability.

### Enforcement Mode
- **D-03:** Start with report-only detection, consistent with `docs/rules/harness.md`.
- **D-04:** Blocking is deferred until false positives and recovery cost are understood.

### Detection Boundary
- **D-05:** Detect first-use code search/project-analysis/impact-analysis commands that use `rg`, `grep`, or `find` before a relevant `mycodemap query|analyze|deps|impact` attempt in the same task/session.
- **D-06:** Allow explicit fallbacks when the task is pure docs/config/log search, CodeMap CLI fails, the file/path is unsupported, or CLI results are insufficient.

### Runtime Coverage
- **D-07:** Claude and Codex are the first runtimes to cover because the audit request and evidence targeted both.
- **D-08:** MCP tool-call compliance is tracked as a separate signal: prior audits saw zero CodeMap MCP tool calls.

### Verification
- **D-09:** Verification must include one violating scenario, one compliant scenario, and one allowed fallback.
- **D-10:** At least one scenario must run as a real subprocess over a real temp repository.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `docs/rules/harness.md` already defines report-only-first escalation and hook design principles.
- `.claude/settings.json` exists but currently has no project-specific CLI-priority hook.
- `.claude/hooks/` contains existing hook scripts that can inform style and packaging.
- Interface contracts expose core commands needed for compliant retrieval paths.

### Established Patterns
- New guardrails should begin as report-only unless they block clear L3/destructive operations.
- Router docs should point to policy authority rather than duplicating full policy text.
- CLI examples should prefer public command name `mycodemap`; `codemap` remains compatibility/internal alias.

### Current Gaps
- `AGENTS.md` Section 6 is human-readable but not yet machine-checkable.
- Claude/Codex runtime sessions can still use `rg` / `grep` / `find` first without any visible warning.
- Prior audit found zero CodeMap MCP tool calls in the inspected windows, so CLI/MCP compliance should remain observable.

</code_context>

<specifics>
## Specific Ideas

- Add a small report-only detector that consumes a tool invocation record and returns a structured warning when `rg` / `grep` / `find` is used for code search before a CodeMap CLI attempt.
- Keep fallback reasons explicit in the warning output so agents can self-correct instead of fighting the guard.
- Document escalation as report-only → measured false-positive review → optional blocking only for low-noise cases.
- For Codex, if there is no native hook surface equivalent, provide an audit script and router documentation rather than pretending enforcement exists.

</specifics>
