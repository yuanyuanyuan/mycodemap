### Pattern 1: Pre-commit 渐进式检查插入

**What:** 在现有 pre-commit 流程中插入新的 warn-only 检查步骤
**When to use:** 新增治理规则但不想立即阻断提交时
**Example:**

```sh
# 现有模式：在 .githooks/pre-commit 中
# Step N: 检查逻辑
echo "Checking for X..."
FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '<pattern>' || true)
if [ -n "$FILES" ]; then
    # warn-only: 输出警告但不 exit 1
    echo "WARNING: <message>"
    # blocking: exit 1
fi
echo "X check passed"
```

**Source:** [VERIFIED: `.githooks/pre-commit` 现有 6 步检查流程]

### Pattern 2: CI PR 级别 warn-only 检查

**What:** 在 CI 工作流中添加仅对 PR 触发的检查步骤，初始 warn-only
**When to use:** 新规则需要团队适应期，先 warn 再渐进升级
**Example:**

```yaml
- name: Verify X in commits
  if: github.event_name == 'pull_request'
  run: |
    # 检查逻辑
    if [ condition ]; then
      echo "WARNING: <message>"
      exit 0  # warn-only: 不阻断 CI
    fi
```

**Source:** [VERIFIED: `.github/workflows/ci-gateway.yml` 中 contract gate 已使用 `if: github.event_name == 'pull_request'` 模式]

## Common Pitfalls

### Pitfall 1: Pre-commit grep 模式列表不完整或过于宽泛

**What goes wrong:** 漏掉真实调用模式导致假阴性，或匹配过多导致假阳性噪音
**Why it happens:** 测试文件的"真实调用"定义因测试类型而异
**How to avoid:** 基于 D-02 确认的模式列表 + 实际代码库验证。当前 101 个测试文件中仅极少数使用真实模式，初始告警量可控。
**Warning signs:** pre-commit 输出大量 WARNING 且多数为假阳性 → 需收窄模式列表

**已验证的模式匹配分析：**

| 模式 | 在代码库中的测试文件命中 | 备注 |
|------|--------------------------|------|
| `StdioServerTransport` | 2 个 MCP 测试文件 | Phase 42 真实 transport 范例 |
| `PassThrough` | 同上 | 与 StdioServerTransport 配套 |
| `spawn` | workflow/git-analyzer 测试 | 真实子进程 |
| `execSync` | ship/publisher 测试等 | 真实子进程 |
| `listen` / `createServer` | 无测试文件命中 | 服务端测试场景保留 |
| `mktemp` | 无测试文件命中 | 隔离环境场景保留 |
| `InMemoryTransport` | 无命中（代码库未使用此模式） | MCP SDK InMemory transport 未在测试中使用 |

[VERIFIED: 通过 `grep -rl` 扫描 src/ 和 tests/ 目录确认]

### Pitfall 2: CI [证据] 标签检查与现有 commit 格式冲突

**What goes wrong:** 12 个现有 FEATURE/BUGFIX commit 都没有 `[证据]` 标签，CI 检查上线后会在 PR summary 中输出告警
**Why it happens:** 规则是新增的，历史 commit 不可能追溯合规
**How to avoid:** D-03 明确 warn-only + 仅检查当前 PR 的 commit range（`base.sha..HEAD`），不影响历史。CI 检查应只扫描 PR 新增的 commit，而非全历史。
**Warning signs:** CI 在无 FEATURE/BUGFIX commit 的 PR 上也触发告警 → 检查 grep 条件

### Pitfall 3: AGENTS.md 修改触发 docs guardrail 失败

**What goes wrong:** `npm run docs:check` 可能检测到 AGENTS.md 的结构性变更（如新增 section）导致交叉引用检查失败
**Why it happens:** docs guardrail 验证文档间引用完整性和一致性
**How to avoid:** 在 AGENTS.md 修改后立即运行 `npm run docs:check`，按需调整 AI 文档中的引用
**Warning signs:** `npm run docs:check` 报告 AGENTS.md 相关的 cross-reference 错误

### Pitfall 4: Phase 41/42 豁免条款表述模糊

**What goes wrong:** 豁免条款不够精确，导致后续对"什么算修改"产生歧义
**Why it happens:** "后续修改必须合规"的边界不好界定（一个 typo fix 是否需要补真实测试？）
**How to avoid:** 在 AGENTS.md Section 8.2 中明确豁免边界——"后续修改"指涉及测试逻辑的改动，纯文档/注释/格式改动不触发合规要求
**Warning signs:** Phase 41/42 维护者对豁免边界提出疑问

### Pitfall 5: Pre-commit 步骤编号不一致

**What goes wrong:** 插入步骤 5a 后，后续注释或文档仍引用旧的步骤编号
**Why it happens:** pre-commit 文件中的注释未同步更新
**How to avoid:** 插入新步骤时更新所有步骤编号注释（5→6, 6→7），或使用 5a 子编号保持兼容
**Warning signs:** Pre-commit 输出信息中的步骤编号与实际执行顺序不一致

### Pre-commit 步骤 5a 插入

```sh
# 插入位置：在 "File headers passed" (约第 209 行) 之后，
# 在 "Checking documentation guardrails" (约第 211 行) 之前

echo "File headers passed"

# 5a. 轻量真实验证检查（warn-only）
# 如果新增/修改了测试文件，检查是否包含真实调用模式
echo "Checking for real-world test patterns..."
STAGED_TEST_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep '\.test\.ts$' || true)
if [ -n "$STAGED_TEST_FILES" ]; then
    MISSING_REAL_PATTERN=0
    for file in $STAGED_TEST_FILES; do
        if [ ! -f "$file" ]; then
            continue
        fi
        if ! grep -qE '(spawn|execSync|execFileSync|mktemp|StdioServerTransport|PassThrough|createServer|listen)' "$file"; then
            echo "WARNING: $file does not appear to contain real-world invocation patterns"
            MISSING_REAL_PATTERN=$((MISSING_REAL_PATTERN + 1))
        fi
    done
    if [ $MISSING_REAL_PATTERN -gt 0 ]; then
        echo ""
        echo "Consider adding integration/E2E tests with real subprocess/transport calls."
        echo "This is a warning only; commit will proceed."
        echo ""
    fi
fi

# 5. 检查文档与 CLI 护栏
DOC_GUARDRAIL_FILES=...
```

### CI 证据验证步骤

```yaml
      - name: Verify real-world evidence in commits
        if: github.event_name == 'pull_request'
        run: |
          COMMITS=$(git log --format='%s' "${{ github.event.pull_request.base.sha }}..HEAD")
          REQUIRE_EVIDENCE=false
          while IFS= read -r line; do
            if echo "$line" | grep -qE '^\[(FEATURE|BUGFIX)\]'; then
              REQUIRE_EVIDENCE=true
            fi
          done <<< "$COMMITS"
          
          if [ "$REQUIRE_EVIDENCE" = "true" ]; then
            if ! echo "$COMMITS" | grep -q '\[证据\]'; then
              echo "WARNING: FEATURE/BUGFIX commits detected but no [证据] tag found."
              echo "Please include evidence of real-world validation."
              echo "Expected format: [证据] path:line or [证据] command output"
              # Warn-only; can graduate to: exit 1
              exit 0
            fi
          fi
          echo "Real-world evidence check passed"
```

**Source:** [VERIFIED: `ci-gateway.yml` 已有 `if: github.event_name == 'pull_request'` 模式；`git log --format='%s'` 与 `base.sha..HEAD` 范围已在 contract gate 步骤中使用]
