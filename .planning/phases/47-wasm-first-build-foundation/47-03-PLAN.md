# Plan 47-03: Benchmark Command and Verification Tests

## Objective
Create `codemap benchmark` command that compares WASM vs Native performance on a target repo. Target: <1s startup penalty for WASM mode on 10K-file repo. Add tests for the fallback mechanism.

## Background
Success criterion #5 requires benchmark capability. Success criterion #6 requires <1s startup penalty. We need a benchmark command that measures and reports this.

## Tasks

### Task 1: Create benchmark command

<read_first>
- src/cli/index.ts (command registration pattern)
- src/cli/commands/doctor.ts (example command structure)
- scripts/run-benchmark.sh (existing benchmark script)
</read_first>

<action>
Create `src/cli/commands/benchmark.ts`:

1. Implement `benchmarkCommand(options)` that:
   - Accepts `--target <path>` (default: current directory)
   - Accepts `--mode <native|wasm|both>` (default: both)
   - Accepts `--iterations <n>` (default: 3)
   - Accepts `--json` for structured output

2. For `--mode both`:
   - Run benchmark with native deps (clear any WASM env vars, set them to '0')
   - Run benchmark with WASM deps (set `CODEMAP_USE_WASM_TREE_SITTER=1`, `CODEMAP_USE_WASM_BETTER_SQLITE3=1`)
   - Compare results

3. Benchmark metrics:
   - `coldStartupMs`: time from process start to first command output
   - `parseInitMs`: time to initialize tree-sitter parser
   - `storageInitMs`: time to initialize SQLite storage
   - `firstFileParseMs`: time to parse first file
   - `totalFiles`: number of files processed
   - `filesPerSecond`: throughput

4. Output format (human):
   ```
   ┌─────────────────────────────────────────────────────────┐
   │  CodeMap Benchmark: WASM vs Native                      │
   ├─────────────────────────────────────────────────────────┤
   │  Target: /path/to/repo (10,247 files)                   │
   ├──────────────────────┬──────────────────────────────────┤
   │  Metric              │  Native    │  WASM    │  Delta   │
   ├──────────────────────┼────────────┼──────────┼──────────┤
   │  Cold Startup        │  450ms     │  520ms   │ +70ms    │
   │  Parser Init         │  120ms     │  180ms   │ +60ms    │
   │  Storage Init        │  50ms      │  55ms    │ +5ms     │
   │  First File Parse    │  15ms      │  22ms    │ +7ms     │
   │  Files/Second        │  245       │  198     │ -47      │
   └──────────────────────┴──────────────────────────────────┘
   
   WASM startup penalty: 70ms (< 1000ms threshold ✓)
   ```

5. Output format (JSON):
   ```json
   {
     "target": "/path/to/repo",
     "fileCount": 10247,
     "native": { "coldStartupMs": 450, "parseInitMs": 120, ... },
     "wasm": { "coldStartupMs": 520, "parseInitMs": 180, ... },
     "penaltyMs": 70,
     "thresholdMs": 1000,
     "passesThreshold": true
   }
   ```

6. Register in `src/cli/index.ts`:
   ```typescript
   import { benchmarkCommand } from './commands/benchmark.js';
   
   program
     .command('benchmark')
     .description('Compare WASM vs Native performance')
     .option('-t, --target <path>', 'Target repository', '.')
     .option('-m, --mode <mode>', 'Benchmark mode: native, wasm, both', 'both')
     .option('-i, --iterations <n>', 'Number of iterations', '3')
     .option('-j, --json', 'JSON output')
     .action(benchmarkCommand);
   ```

7. Add [META] and [WHY] comments.
</action>

- src/infrastructure/storage/adapters/sqlite-loader.ts (after Plan 47-01)

<action>
Create `src/cli/__tests__/wasm-fallback.test.ts`:

1. Test `checkAndActivateWasmFallback`:
   - When native module available: returns nativeAvailable=true, fallbackActivated=false
   - When native unavailable and WASM available: returns fallbackActivated=true, sets env var
   - When both unavailable: returns actionable error with remediation

2. Test `tree-sitter-loader`:
   - When `CODEMAP_USE_WASM_TREE_SITTER='1'`: loads web-tree-sitter (or mocks it)
   - When env var not set: loads native tree-sitter
   - Returns compatible Parser API in both paths

3. Test `sqlite-loader`:
   - When `CODEMAP_USE_WASM_BETTER_SQLITE3='1'`: loads node:sqlite or sql.js
   - When env var not set: loads better-sqlite3
   - Returns compatible SQLiteDatabaseLike interface in both paths

4. Use mocking for unavailable modules (vitest `vi.mock`):
   ```typescript
   vi.mock('tree-sitter', () => ({ default: class MockParser {} }));
   vi.mock('web-tree-sitter', () => ({ default: class MockWasmParser {} }));
   ```

5. Clean up env vars in `beforeEach`/`afterEach` to avoid test pollution.
</action>
