<objective>
Relocate `TreeSitterParser` from the legacy `src/parser/` layer into `src/infrastructure/parser/` as a proper `ILanguageParser` implementation via `ParserBase`, register it in `ParserRegistry`, and decide the fate of the regex-based `TypeScriptParser`.
</objective>

<must_haves>
- A `TreeSitterParser` class exists in `src/infrastructure/parser/implementations/` extending `ParserBase` and implementing `ILanguageParser`.
- It is registered in `createDefaultParserRegistry()` for TypeScript/JavaScript files.
- The legacy `src/parser/implementations/tree-sitter-parser.ts` is either removed or reduced to a re-export stub with deprecation warning.
- `tree-sitter-loader.ts` remains functional (may be moved or kept in place if relative paths are stable).
- No regression in `mycodemap generate` output for TypeScript projects.
</must_haves>

<tasks>
<task id="1" name="Create infrastructure TreeSitterParser extending ParserBase">
<description>
Create `src/infrastructure/parser/implementations/TreeSitterParser.ts`:
1. Extend `ParserBase`.
2. Implement `languageId = 'typescript'` and `fileExtensions = ['ts', 'tsx', 'js', 'jsx', 'mjs', 'cjs']`.
3. In `doInitialize()`, call `loadTreeSitter()` (from `src/parser/implementations/tree-sitter-loader.ts` or moved copy).
4. In `parseFile(filePath, content, options)`, use the loaded tree-sitter Parser to parse `content` and extract:
   - `module` (with stats)
   - `symbols` (classes, functions, interfaces, etc.)
   - `imports` and `exports`
   - `dependencies` (resolved from imports)
   - optional `callGraph` and `complexity` if options request them
5. Return a registry-shaped `ParseResult` directly.
6. Reuse AST traversal logic from the legacy `TreeSitterParser` where possible, but adapt it to the `ILanguageParser` contract.
</description>
<verify>
- `rtk npx tsc --noEmit`
- New file passes lint
- `mycodemap query -s "TreeSitterParser"` shows the new infrastructure location
</verify>
</task>

<task id="2" name="Register TreeSitterParser in createDefaultParserRegistry">
<description>
In `src/infrastructure/parser/index.ts`:
1. Import the new `TreeSitterParser`.
2. Update `createDefaultParserRegistry()` to register `TreeSitterParser` for TS/JS files.
3. Decide on regex `TypeScriptParser` fate:
   - **Option A (Recommended):** Remove regex `TypeScriptParser` registration entirely; rely on TreeSitterParser + WASM fallback.
   - **Option B:** Keep regex `TypeScriptParser` registered under a different `languageId` (e.g., `typescript-regex`) for explicit fallback.
   - **Option C:** Keep regex `TypeScriptParser` but demote it — register only if TreeSitter initialization fails.
4. Document the decision in code comments.
</description>
<verify>
- `rtk rg "new TreeSitterParser" src/infrastructure/parser/index.ts` returns a match.
- `mycodemap generate` on a TS repo completes without error.
- `rtk ./node_modules/.bin/vitest run src/infrastructure/parser/__tests__` (or create minimal test)
</verify>
</task>
