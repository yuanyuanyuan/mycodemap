# Phase 999.1: mycodemap init enhancements — Context

**Gathered:** 2026-04-20
**Status:** Ready for planning

<domain>
## Phase Boundary

将 mycodemap 的 `init` 命令从"在项目根目录创建配置文件"升级为"项目级 AI 助手基础设施初始化"。交付物是增强后的 `mycodemap init` 命令，它会把所有 mycodemap 相关文件收敛到 `.mycodemap/` 目录内，并引导用户安装 git hooks 和 AI 护栏规则。

**In scope:**
- 配置文件的集中化（从根目录 → `.mycodemap/`）
- 向后兼容的自动迁移（根目录旧配置 → `.mycodemap/`）
- `.mycodemap/` 内部子目录结构的统一（logs/、workflow/、rules/ 等）
- git hooks 的检测、安装和同步
- AI 护栏规则文件的生成、同步和注入引导
- init 命令的交互式用户引导

**Out of scope:**
- 修改非 init 命令的路径解析逻辑（generate/query 等仍使用现有路径解析）
- 为 AI 规则创建新的 CLI 子命令（如 `mycodemap sync-rules`）——仅讨论中提及，不属于本 phase 范围
- 修改非 mycodemap 相关的项目文件（如 package.json）
- 自动修改用户项目的 AI 上下文文件（CLAUDE.md / AGENTS.md）——仅提示用户手动操作
</domain>

<decisions>
## Implementation Decisions

### 工作空间路径策略
- **D-01:** 配置文件收到 `.mycodemap/config.json`（简化为 `config.json`，而非保留 `mycodemap.config.json`）
- **D-02:** 发现根目录旧配置时**自动迁移**（复制内容到 `.mycodemap/config.json`，然后提示用户删除旧文件），不采用兼容读取模式
- **D-03:** 全部 mycodemap 相关文件统一收进 `.mycodemap/` 下的子目录（logs/、workflow/、rules/、storage/ 等），不使用扁平结构
- **D-04:** 清理旧的 `.codemap` → `.mycodemap` 迁移提示，只保留新的根目录配置迁移提示

### Git hooks 安装策略
- **D-05:** hooks 安装方式为**复制到目标项目的 `.git/hooks/`**，不使用 `core.hooksPath`
- **D-06:** 安装前检测目标项目已有 hooks，提示用户选择：覆盖、跳过或合并
- **D-07:** 内置 hooks 内容为**完整复制** CodeMap 自身 `.githooks/` 下的 `pre-commit` 和 `commit-msg`
- **D-08:** 每次运行 `mycodemap init` 时检查 hooks 同步状态，如有变化提示用户更新

### AI 护栏规则的内容与注入
- **D-09:** 规则模板采用**内置精简规则集**（通用型，不依赖项目类型），不从 `docs/rules/` 动态生成
- **D-10:** 规则文件组织为**分类子目录**结构：`.mycodemap/rules/commit/`、`.mycodemap/rules/test/`、`.mycodemap/rules/lint/` 等
- **D-11:** 规则注入方式为**提示用户手动添加引用**到 `CLAUDE.md` 或 `AGENTS.md`，不自动修改用户文件
- **D-12:** 每次运行 `mycodemap init` 时检查规则同步状态，如有变化提示用户更新

### init 命令的交互模式
- **D-13:** 三个子功能（workspace 集中化、hooks 安装、rules 注入）以**一键选择**方式呈现，用户可以一次性勾选要启用的功能
- **D-14:** 扩展 `first-run-guide.ts` 的欢迎信息，在现有 "init → generate → help" 之后增加 "设置 git hooks" 和 "配置 AI 护栏规则" 的步骤说明

### Claude's Discretion
- 子目录的具体命名（logs/ 还是 log/、rules/ 还是 guardrails/）——遵循现有代码中的命名惯例
- 精简规则集的具体内容（commit/test/lint 等类别下的具体规则条目）——参考 CodeMap 自身 docs/rules/ 的内容提炼
- 交互式提示的措辞和 CLI 输出格式——遵循现有 CLI 的风格（chalk 颜色、emoji、中文）
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/cli/paths.ts` — `resolveConfigPath()` 和 `resolveOutputDir()` 已有新旧路径兼容逻辑，可扩展为支持 `.mycodemap/config.json`
- `src/cli/config-loader.ts` — `createDefaultCodemapConfigFile()` 生成默认配置，迁移后需要更新输出路径
- `src/cli/first-run-guide.ts` — `showFirstRunGuide()` 和 `markFirstRunDone()` 可直接扩展

### Established Patterns
- 新旧兼容：CodeMap 已有 `.codemap` → `.mycodemap` 的迁移经验（paths.ts），新迁移可复用相同模式
- CLI 交互：使用 `chalk` 颜色 + emoji + 中文输出，保持风格一致
- 错误处理：命令入口使用 `try/catch` + `process.exit(1)`

### Integration Points
- `src/cli/index.ts` — 需要注册新的 init 选项（如 `--interactive`）
- `.git/` — hooks 复制目标目录
- `CLAUDE.md` / `AGENTS.md` — 规则引用目标文件（只提示，不修改）
</code_context>

<specifics>
## Specific Ideas

- init 流程的输出应该像 "脚手架安装器" 一样清晰，每一步都有明确的完成标记
- hooks 的冲突检测应该比较文件内容哈希，而不仅仅是文件名存在
- 规则文件的更新提示应该显示 diff 预览，让用户知道变化了什么
- 欢迎信息的扩展应该保持简洁，不要变成冗长的教程
</specifics>
