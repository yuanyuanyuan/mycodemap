# POC Validator — 真实场景可行性验证技能

> 验证一个新 feature / API / 设计方案是否真的可行，而非"看起来可行"。
>
> 核心原则：**纯 mock 测试 ≠ 真实可行。必须用真实数据、真实环境、真实调用来验证。**

## 触发条件

以下场景必须激活本技能：
- 用户说"做个 POC"、"验证一下可行性"、"确认这个设计能跑通"
- 涉及第三方 API 集成（Stripe、OpenAI、AWS、支付网关等）
- 涉及内部 API / 微服务 / MCP server 调用
- 涉及子代理行为改变、规则注入/检索、上下文传递
- 涉及端到端数据流（CLI → 文件系统 → 输出 / 前端 → 后端 → 数据库）
- 任何需要"真实场景验证"的设计（AGENTS.md Section 8.1 范畴）
- 需要确认"在真实数据下性能是否可接受"

## 核心原则：四真原则

| 原则 | 含义 | 反例（必须避免） |
|------|------|----------------|
| **真实环境** | 用 sandbox / staging / 隔离副本，不用 mock server | `jest.mock()`、`nock`、本地伪造的 HTTP server |
| **真实数据** | 用真实请求参数、真实文件、真实数据库记录 | 硬编码 fixture、只有 3 条测试数据 |
| **真实调用** | 用真实子进程、真实网络请求、真实 Agent spawn | 只测函数返回值、只断言 mock 被调用 |
| **真实失败** | 至少验证一个真实失败场景 | 只跑 happy path |

## POC 类型与验证策略

根据被验证对象的类型，选择对应的验证策略：

### 类型 A：API 集成型 POC

**场景**：接入第三方 API（支付、AI、地图、短信）、调用内部微服务、MCP server。

**验证手段**：
1. **真实调用**：用真实 API key / token 向 sandbox / 测试环境发送请求
2. **响应验证**：检查 HTTP status、响应体结构、关键字段类型
3. **错误验证**：故意传错参数、用错误 credential、触发 rate limit，观察错误响应
4. **幂等/重试**：同一请求发两次，观察是否幂等

**负面场景（至少选一个）**：
- 错误 API key → 应返回 401/403，不应 crash
- 超时 / 断网 → 应有明确的超时处理和错误信息
- 非法参数 → 应返回 400 并带具体错误说明
- Rate limit → 应能识别 429 并支持退避重试

**判定标准示例**：
```
✅ 向 Stripe test env 创建 payment intent 成功，返回 client_secret
✅ 用错误 key 调用返回 401，错误信息可读
✅ 同一 idempotency key 调用两次只创建一个 intent
❌ 超时未处理，hang 住 30 秒
```

---

### 类型 B：子代理行为型 POC

**场景**：AI 代理规则系统、上下文传递、检索/注入机制、工具调用链。

**验证手段**：
1. **真实子代理**：用 `Agent` 工具 spawn 真实子代理实例（非 mock）
2. **行为对比**：跑对照组（有检索指引 vs 无检索指引）
3. **口头报告**：要求子代理报告决策理由，确认规则进入了决策过程
4. **负面验证**：规则缺失时，子代理是否拒绝盲目操作

**实验设计模板**：

| 实验 | 目的 | 设计要点 |
|------|------|---------|
| **功能可用** | CLI/API 能输出正确结果 | 直接调用，检查 JSON 结构和字段 |
| **实验组** | 有检索指引时子代理是否遵守规则 | Prompt 中明确要求"先查规则再干活" |
| **对照组** | 排除"子代理本来就知道"的可能性 | 同样的任务，但禁止主动查规则 |
| **负面验证** | 无规则时子代理是否报告失败 | 空/半空目录，观察元认知反应 |

**判定标准示例**：
```
✅ 实验组使用 [TAG] scope: message 格式，对照组遗漏 rtk 规则
✅ 子代理明确报告"No confidence to operate blindly"
❌ 子代理查到规则但继续用 npm test（未遵守）
❌ 无规则时子代理假装知道并继续执行
```

---

### 类型 C：数据流/端到端型 POC

**场景**：CLI 工具链、文件处理管道、构建流程、前后端数据流。

**验证手段**：
1. **真实文件系统**：在 `/tmp` 创建真实文件，验证读写、权限、编码
2. **真实子进程**：`child_process.spawn` 或 Shell 执行真实命令
3. **端到端链路**：验证输入 → 处理 → 输出的完整链条
4. **中间状态可见**：在关键节点打印日志，确认数据 transformations 正确

**负面场景（至少选一个）**：
- 输入文件不存在 / 格式错误 → 应有清晰错误，不应写脏数据
- 中间步骤失败 → 应回滚或报错，不应残留临时文件
- 大文件 / 边界数据 → 应正确处理，不应内存溢出

**判定标准示例**：
```
✅ 1000 行 CSV 输入 → 处理 → 输出 JSON，字段无丢失
✅ 输入文件不存在时返回 1 并打印"File not found"
✅ 处理中断后 /tmp 无残留临时文件
❌ 中文字段出现乱码
❌ 大文件导致 OOM
```

---

### 类型 D：性能型 POC

**场景**：大数据量处理、高并发、实时计算、复杂算法。

**验证手段**：
1. **真实数据量**：用生产级数据量（或按比例缩小但仍具代表性）
2. **真实硬件**：在目标部署环境（或等效环境）上跑
3. **可观测指标**：记录 latency、p99、内存峰值、CPU 占用
4. **渐进压力**：从小数据量逐步加压，找到拐点

**负面场景（至少选一个）**：
- 数据量翻倍 → latency 是否线性增长？
- 并发 10/50/100 → 是否出现锁竞争或连接池耗尽？
- 长时间运行 → 是否有内存泄漏？

**判定标准示例**：
```
✅ 10k 条记录处理 < 2s，内存 < 512MB
✅ 并发 50 时 p99 < 500ms，无 5xx
⚠️ 100k 条记录时 latency 上升到 15s，需要优化或分页
❌ 内存随时间线性增长（泄漏）
```

## 通用执行步骤

### Step 1: 环境隔离（Replicate）

```bash
# 必须复制到 /tmp 或隔离目录，禁止直接修改源码
cp -r /data/project /tmp/project-poc
```

- 如果涉及构建，在隔离副本中 `npm run build`
- 如果涉及数据库，使用 docker 临时实例或 sandbox 环境
- 如果涉及 API key，使用 test/sandbox key（**禁止用生产 key**）

### Step 2: 功能可用性验证（Function）

在隔离环境中直接调用目标功能，确认最基础的 happy path 能跑通：

```bash
cd /tmp/project-poc
node dist/cli/index.js <new-command>
curl -s http://localhost:3000/api/health
```

**验收标准**：
- [ ] 命令/API 不 crash
- [ ] 输出是机器可解析的（JSON / 结构化文本 / HTTP 响应）
- [ ] 输出内容符合设计文档的数据模型

### Step 3: 真实数据验证（Real Data）

**替换所有 mock**：
- API 型：用真实 HTTP 请求替代 `jest.mock('axios')`
- 数据流型：用真实 1000+ 行文件替代 3 条 fixture
- 子代理型：用真实 `Agent` spawn 替代假想的"子代理会这样做"

**验证记录**：
- 保留真实请求/响应的日志（脱敏后）
- 保留真实子代理的 tool call 记录
- 保留真实文件处理前后的 diff

### Step 4: 失败场景验证（Failure）

**必须设计至少一个失败场景**，证明系统不会静默崩溃：

| POC 类型 | 失败场景示例 |
|---------|------------|
| API 集成 | 错误 credential、超时、非法参数、rate limit |
| 子代理行为 | 规则缺失、工具调用失败、上下文截断 |
| 数据流 | 文件不存在、格式错误、中间步骤 crash、权限不足 |
| 性能 | 数据量翻倍、并发超限、长时间运行 |

**判定标准**：
- [ ] 失败时系统有明确的错误信息（不 hang、不吞异常）
- [ ] 失败后系统状态可预期（无脏数据、无残留资源）

### Step 5: 对照/基准验证（Baseline）

**目的**：证明新方案比旧方案好，或证明某个机制有增量价值。

| POC 类型 | 对照设计 |
|---------|---------|
| API 集成 | 旧 SDK vs 新 SDK 的延迟/错误率对比 |
| 子代理行为 | 有检索指引 vs 无检索指引的规则覆盖数对比 |
| 数据流 | 旧流程 vs 新流程的处理时间/正确率对比 |
| 性能 | 优化前 vs 优化后的基准数据对比 |

**没有对照组，就无法证明 POC 的价值。**

### Step 6: 证据汇总与可信度自评

必须输出以下结构：

```markdown
## 实验结果汇总

| 实验 | 假设 | 验证方式 | 结果 | 状态 |
|------|------|---------|------|------|
| A: 功能可用 | 命令/API 能正常输出 | 直接调用 | ... | ✅/❌ |
| B: 真实数据 | 真实数据下结果正确 | 真实请求/文件/Agent | ... | ✅/❌ |
| C: 失败场景 | 失败时系统不崩溃 | 故意触发错误 | ... | ✅/❌ |
| D: 对照验证 | 新方案优于基线 | 对比实验 | ... | ✅/❌ |

## 关键证据
- [证据] 真实请求日志：...
- [证据] 真实响应数据：...
- [证据] 子代理 tool call 记录：...
- [证据] 性能指标截图：...

## 可信度自评
- **确定信息**（基于真实验证）：...
- **推测信息**（基于模式匹配）：...
- **需验证信息**（未在 POC 中覆盖）：...
- **风险标记**：...
```

## 常见反模式（必须避免）

| 反模式 | 错误示例 | 为什么错 |
|--------|---------|---------|
| **纯 mock 验证** | "`jest.mock` 后断言函数返回了正确对象，POC 通过" | mock 掩盖了真实网络的序列化问题、时区问题、编码问题 |
| **只看 happy path** | "API 返回 200，POC 通过" | 没验证错误 credential、超时、rate limit |
| **数据量太小** | "用 3 条 fixture 数据验证 CSV 解析" | 无法发现大文件的内存问题、特殊字符问题 |
| **无对照组** | 只跑新方案，不跑旧方案/基线 | 无法证明新方案真的有优势 |
| **在源码目录直接改** | 不复制到 `/tmp` | 污染工作树，可能触发 hooks，且无法隔离对比 |
| **观察指标模糊** | "看起来快了"、"子代理看起来遵守了" | 必须有可量化的指标（延迟 ms、commit 格式精确匹配） |

## 验收清单（DoD）

POC 完成前必须逐条确认：

- [ ] **源码未污染**：所有修改在 `/tmp` 隔离环境
- [ ] **真实环境**：使用了 sandbox / staging / 隔离实例，非本地 mock
- [ ] **真实数据**：用了真实请求/文件/记录，非硬编码 fixture
- [ ] **真实调用**：用了真实子进程/网络请求/Agent spawn，非函数级单元测试
- [ ] **失败场景**：至少验证了一个真实失败路径
- [ ] **对照验证**：有对比实验（新 vs 旧、有 vs 无）
- [ ] **可判定标准**：每个实验有明确的通过/不通过判定依据
- [ ] **证据留存**：保留了真实请求日志、响应数据、tool call 记录（脱敏后）
- [ ] **临时文件已清理**：`rm -rf /tmp/project-poc`

## 示例参考

### 示例 1：Phase 58 — 子代理行为型 POC

详见 `.planning/phases/58-subagent-environment-contract-injection/`。

- **类型**：子代理行为型
- **真实验证**：用 `Agent` 工具 spawn 真实子代理，执行真实 Bash 命令
- **真实数据**：项目仓库的真实规则文件（AGENTS.md、.githooks、docs/rules）
- **对照设计**：实验组（有检索指引）vs 对照组（无检索指引）
- **失败验证**：空目录场景下子代理拒绝盲目操作
- **关键证据**：实验组发现 12 项规则，对照组仅发现 ~4 项；rtk 规则、验证阈值对照组全部遗漏

### 示例 2：第三方支付 API 集成 POC（假想）

- **类型**：API 集成型
- **真实验证**：向 Stripe test environment 发送真实 HTTP 请求
- **真实数据**：使用 Stripe 提供的 test card numbers（`4242 4242 4242 4242`）
- **对照设计**：旧 SDK 调用方式 vs 新封装层的延迟对比
- **失败验证**：错误 API key → 401；非法金额 → 400；超时 → 10s 后返回明确错误
- **关键证据**：真实 payment intent 创建日志、错误响应结构、幂等性验证记录

## 技能元数据

- **适用范围**：API 集成、AI 代理工作流、CLI 工具链、数据流管道、性能基准
- **不适用**：纯算法数学证明、纯 UI 视觉走查、已上线功能的回归测试
- **前置依赖**：被验证对象可构建/可部署、有 sandbox 或隔离环境可用
- **预期耗时**：30-90 分钟（取决于 POC 类型和需要调用的外部服务数量）
