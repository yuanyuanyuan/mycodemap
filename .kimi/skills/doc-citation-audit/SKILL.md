# 文档引用一致性检查（Doc Citation Audit）

> **版本**: 1.0  
> **来源**: Phase 58 文档验证实践总结  
> **核心原则**: 文档中的每一个事实声明都必须能被追溯到原始来源，推论必须与事实分开标注。

---

## 触发条件

当遇到以下任一情况时，**必须**执行此技能：

1. 文档声称"经官方文档验证/确认/交叉验证"
2. 文档引用了外部 issue、PR、论文、规范文档
3. 文档把代码内部变量名、私有 API、未公开机制当作公共能力引用
4. 文档混用了不同来源的推论（官方文档 ↔ 社区 issue ↔ 个人推论）
5. 用户问"你有证据吗""这个结论怎么来的""引用准确吗"
6. 设计文档从 v1 升级到 v2，需要核对旧版本引用的有效性

---

## 核心方法：六步验证法

### Step 1: 建立引用清单（Citation Inventory）

把文档中提到的**所有**外部引用列成一张表，不遗漏、不合并。

**扫描范围：**
- 超链接（`[text](url)`）
- Issue/PR 编号（`#12345`）
- 章节引用（"官方文档 Section X"）
- 隐性引用（"社区验证表明""经交叉验证"等未附链接的声明）

**输出格式：**

```markdown
| 引用标识 | 出现位置（文档行号） | 声称的内容 | 声称的来源 |
|---------|-------------------|-----------|-----------|
```

**🛑 检查点 1**：完成引用清单后，先向用户展示完整清单并确认「是否有遗漏的引用或隐性声明？」，用户确认后再进入 Step 2。

### Step 2: 双向核对（Bidirectional Cross-Check）

**三个清单必须一致：**
- **清单 A**: 正文提到的引用
- **清单 B**: 参考资料/Canonical References 列出的引用
- **清单 C**: 实际访问原始来源后确认存在的内容

**常见问题：**
- 正文提到但参考资料列表遗漏（如 Phase 58 的 #11004）
- 参考资料列表有但正文未使用（死引用）
- 链接指向错误的目标（如把 blog 链接当成官方文档）

### Step 3: 来源分级（Source Grading）

每个事实声明必须标注来源等级：

| 等级 | 标识 | 说明 | 可信度 |
|------|------|------|--------|
| **L1** | `[官方]` | 官方文档、API 规范、源代码注释中明确描述的行为 | 高 |
| **L2** | `[社区]` | GitHub issue、社区讨论、第三方博客中的报告 | 中（需标注 open/closed 状态） |
| **L3** | `[推论]` | 基于 L1/L2 的逻辑推导，原始来源未直接确认 | 低（必须标注推导路径） |
| **L4** | `[假设]` | 无直接来源，基于经验或模式匹配的猜测 | 必须显式声明 |

**红线规则：**
- L2/L3/L4 **不得**伪装成 L1
- 禁止写"官方文档确认 XX 有缺陷"——官方文档不会确认自己的缺陷
- 禁止写"经官方文档交叉验证，假设不成立"——官方文档只描述机制，不验证假设

**🛑 检查点 2**：完成来源分级后，向用户展示「来源分级修正表」，确认修正方向（尤其是 L1→L2/L3 的降级）无误后再继续。

### Step 4: 术语审计（Terminology Audit）

检查文档中使用的技术术语是否在原始来源中存在。

**常见陷阱：**
- 把代码内部变量名当作公共 API 名称引用（如 `SPAWN_AGENT_DEVELOPER_INSTRUCTIONS`）
- 把 issue 作者的推测当作官方术语（如 issue 说"看起来像 bug"，文档写成"官方确认的 bug"）
- 把私有/实验性功能当作稳定功能引用（如未公开的 hook 参数）

**检查方法：**
1. 在原始来源页面搜索该术语
2. 如果来源是代码仓库，搜索代码中是否有该字符串
3. 如果来源是文档，确认该术语出现在公共 API 或文档中，而非内部实现

### Step 5: 标注缺口（Gap Annotation）

明确标注"官方文档**没说什么**"，和"社区报告了什么"。

**模板：**
```markdown
**官方文档确认：**
- ✅ 机制 A 存在（章节 X）
- ✅ 机制 B 的格式为 Y（章节 Z）

**官方文档未提及：**
- ❌ 机制 A 的已知缺陷
- ❌ 机制 B 在 Windows 上的行为差异

**社区 issue 报告：**
- ⚠️ #12345 — 机制 A 在场景 X 下失效
- ⚠️ #12346 — 机制 B 的内部变量无条件注入
```

**🛑 检查点 3**：输出缺口标注后，询问用户「是否还有其他已知信息或内部上下文需要补充？」，确认后再进入 Step 6 生成最终报告。

### Step 6: 输出报告（Audit Report）

**报告结构：**

```markdown
## 文档引用一致性审计报告

**审计对象**: `<文档路径>`
**审计时间**: `<时间>`
**审计人**: `<who>`

### 1. 引用清单（共 N 个）
...

### 2. 双向核对结果
- ✅ 一致: N 个
- ⚠️ 正文有但列表缺: N 个（如 #11004）
- ⚠️ 列表有但正文未用: N 个
- ❌ 链接失效: N 个

### 3. 来源分级修正
| 原文声称 | 实际等级 | 修正建议 |
|---------|---------|---------|
| "官方文档确认 XX 有缺陷" | L2（社区 issue） | 改为"社区 issue #N 报告 XX 有缺陷，官方文档未提及" |

### 4. 术语审计
- 发现非官方术语: `SPAWN_AGENT_DEVELOPER_INSTRUCTIONS`
- 修正为: "issue #17323 报告的无条件 `<spawned_agent_context>` prompt 块"

### 5. 缺口标注
...

### 6. 待验证项
- 需要本地 PoC 验证: ...
- 需要向官方确认: ...
```

---

## 快速检查清单（Quick Checklist）

执行完六步验证后，逐项确认：

- [ ] 文档中所有外部链接都访问过，没有 404
- [ ] 所有 issue/PR 编号都确认存在，标注了 open/closed 状态
- [ ] 正文提到的引用和参考资料列表完全一致（无遗漏、无死引用）
- [ ] 每个事实声明都标注了来源等级（L1/L2/L3/L4）
- [ ] 没有 L2/L3/L4 伪装成 L1 的情况
- [ ] 所有技术术语都能在原始来源中找到对应
- [ ] 代码内部变量名/私有 API 没有当作公共机制引用
- [ ] "官方文档未提及"的内容被明确列出
- [ ] 文档中的"已确认""已验证"等强断言都有对应来源支撑
- [ ] 如果存在"推论"，说明了推导路径（从哪些证据得出）

## 真实案例：Phase 58 文档验证

### 背景
Phase 58 设计文档声称"经与社区实践和官方文档交叉验证后，假设存在根本性缺陷"。用户质疑证据，触发本技能。

### 执行过程

**Step 1 — 引用清单：**
列出 9 个引用（8 个 issue + 1 个官方文档）。

**Step 2 — 双向核对：**
发现正文提到 Codex #11004，但 canonical refs 列表中遗漏。

**Step 3 — 来源分级：**
| 原文 | 声称等级 | 实际等级 | 问题 |
|------|---------|---------|------|
| "经官方文档交叉验证后，假设有缺陷" | L1 | L2 | 官方文档不验证缺陷 |
| "additionalContext 在压缩时会被丢弃" | L1（issue 已确认） | L3（推论） | issue 原文是 may be dropped |
| "SPAWN_AGENT_DEVELOPER_INSTRUCTIONS 为硬编码无条件注入" | L1（平台机制） | L2（issue 报告的内部变量） | 官方文档无此术语 |

**Step 4 — 术语审计：**
- `SPAWN_AGENT_DEVELOPER_INSTRUCTIONS` 在官方文档中搜索不到。
- issue #17323 原文描述为无条件附加 `<spawned_agent_context>` prompt 块。

**Step 5 — 缺口标注：**
官方文档确认：built-in agents、`.toml` 格式、`developer_instructions` 必填。
官方文档未提及：无条件 prompt 注入、Windows bug、项目级 config 不生效。

**Step 6 — 修正输出：**
- 去掉"官方文档交叉验证"的说法
- #23885 改为"可能丢弃"
- `SPAWN_AGENT_DEVELOPER_INSTRUCTIONS` 替换为 issue 原文描述
- 新增验证记录章节

## 常见错误模式

| 错误模式 | 示例 | 修正 |
|---------|------|------|
| **来源攀附** | "经官方文档验证，XX 有 bug" | "官方文档确认 XX 机制存在；社区 issue #N 报告 XX 有 bug" |
| **推论硬化** | "issue 确认 YY 会被丢弃" | "issue 报告 YY may be dropped（可能性）" |
| **术语冒用** | "使用 `INTERNAL_VAR` 配置" | "issue #N 报告代码中 `INTERNAL_VAR` 的行为，官方文档未公开此配置" |
| **清单遗漏** | 正文引用 #123，refs 列表没有 | 补上 #123，或从正文移除 |
| **状态未更新** | 引用已关闭的 issue 作为开放证据 | 核实 issue 状态，如已关闭需确认修复版本 |

## 工具推荐

| 场景 | 工具 | 用法 |
|------|------|------|
| 检查链接有效性 | `curl -I <url>` 或 `FetchURL` | 批量验证引用链接 |
| 搜索术语出处 | `rg <term> <repo>` 或 `Grep` | 确认术语是否存在于官方仓库 |
| 提取文档引用 | `rg '\[.*?\]\(.*?\)|#\d{4,}' <doc>` | 快速列出所有链接和 issue 编号 |
| 对比两个清单 | `comm` 或 `diff` | 对比正文引用和 refs 列表 |

---

## 输出承诺

执行此技能后，必须交付：
1. 引用清单（所有外部引用的完整列表）
2. 双向核对结果（遗漏、死引用、失效链接）
3. 来源分级修正表（哪些声明需要降级）
4. 术语审计结果（哪些非官方术语需要替换）
5. 缺口标注（官方文档确认了/未提及/社区报告了什么）
6. 待验证项（哪些结论需要 PoC 或进一步确认）

**底线原则：如果无法验证，必须标注为 `[假设]` 或 `[未验证]`，不得伪装成事实。**
