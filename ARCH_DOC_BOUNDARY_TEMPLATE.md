# 架构文档分层模板（1页）

> 用途：明确“概要设计保留什么、详细设计下沉什么”，避免过度设计或信息缺失。
> 适用：CodeMap 重构相关文档（Overview + 各模块 Detailed Design）。

## A. 概要设计应保留项（1页模板）

### 1) 目标与边界（必须）
- **业务目标**：一句话说明为什么做（例：提升 AI 与人类检索效率）。
- **范围 In-Scope**：本期明确支持的能力/语言/入口。
- **非目标 Out-of-Scope**：明确不做什么，防止需求漂移。

### 2) 全局约束（必须）
- **硬约束**：性能、兼容性、安全、规范（如工具禁用/必须项）。
- **冲突裁决规则**：当文档冲突时，以哪份文档/规则为准。

### 3) 架构与职责（必须）
- **系统上下文图**：上游输入 → 编排层 → 工具层 → 输出层。
- **模块职责表**：每个模块一句话职责 + 输入/输出摘要。
- **关键决策**：为什么选这个方案（1~3 条）。

### 4) 接口摘要契约（必须）
- **CLI/API 摘要**：参数集合、最小合法值、默认策略（不写实现细节）。
- **输出摘要**：是否有 `schemaVersion`、稳定字段列表、兼容策略一句话。

### 5) 验收与里程碑（必须）
- **可复验 KPI**：指标、阈值、测量入口（脚本/数据集路径链接）。
- **分阶段 DoD**：每阶段进入条件/退出条件（简版）。

### 6) 风险与回退（必须）
- **Top 风险**：至少 3 条（技术/交付/质量）。
- **失败预演**：至少 1 个真实失败模式 + 回退方案。

### 7) 文档索引（必须）
- **详细设计目录**：每个模块对应详细文档链接。
- **追踪矩阵**：概要章节 → 详细章节映射（防止双真相）。

---

## B. 详细设计应下沉项（清单）

> 原则：概要给“方向和约束”，详细给“可实现和可测试”。

### 1) 算法与规则细节
- [ ] 路由规则（intent 枚举、优先级、冲突处理）。
- [ ] 评分/置信度公式（权重、阈值、样例计算）。
- [ ] 排序与融合细则（去重键、稳定排序、并列规则）。

### 2) 数据与契约细节
- [ ] 完整 Schema（JSON Schema/TS types）。
- [ ] 字段语义、必填/可选、默认值、边界值。
- [ ] 版本策略（`schemaVersion`、向后兼容、弃用流程）。

### 3) 执行与异常流
- [ ] 时序图/状态机（正常流、降级流、失败流）。
- [ ] 超时、重试、熔断、回退链路规则。
- [ ] 错误码表（触发条件、用户提示、可观测字段）。

### 4) 工程实现
- [ ] 文件结构与模块边界（具体到目录/文件）。
- [ ] 关键接口签名与依赖注入方式。
- [ ] 配置项说明（环境变量、开关、默认值）。

### 5) 测试与基准
- [ ] 单测/集成测/回归测清单与覆盖目标。
- [ ] Golden cases 与反例集合。
- [ ] Benchmark 协议（数据集、脚本、统计口径、固定版本）。

### 6) 运维与观测
- [ ] 日志结构（字段、采样、隐私约束）。
- [ ] 指标与告警（SLO、阈值、看板）。
- [ ] 变更发布与回滚步骤（灰度、开关、恢复时间）。

---

## C. 快速判定规则（写概要还是写详细）

- **放概要**：影响全局决策、跨模块协同、验收口径、规范一致性。
- **放详细**：需要“照着就能编码/测试/运维”的实现信息。
- **冲突处理**：若同一信息在两层重复，以详细为实现真相；概要仅保留摘要与链接。

---

## D. 推荐落地方式（针对当前项目）

1. 先在概要补齐“最小可执行约束”：范围、硬约束、接口摘要、指标测量入口。
2. 把公式/阈值/错误码/Schema 全量下沉到对应详细文档。
3. 增加“概要→详细”映射表，避免后续文档漂移。
